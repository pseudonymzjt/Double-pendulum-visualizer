# 我遇到的困难 — 混沌艺术：双摆可视化器

> 开发过程中踩过的坑、撞过的墙、以及最后怎么爬出来的。一共 18 个，按时间顺序排列，从最早的 Verlet 阶段一直到最后做性能优化的时候。

---

## 1. Verlet 重力太弱——摆冻住了

**背景**：第七阶段，我把物理引擎从基于角度的 RK4 换成了基于粒子的 Verlet 积分。Verlet 的好处是天然支持 N 节链——每个粒子存 `{x, y, px, py}`，约束求解器负责保持杆长。理论上很优雅。

**有多疼**：中。摆看起来冻住了。

**发生了什么**：代码写完之后，我打开页面，摆锤几乎纹丝不动。我以为代码写错了，检查了重力方向，检查了 dt，检查了每个粒子的质量——看起来都对。但摆就是不动。

**为什么**：Verlet 重力的计算公式是 `y += vy + G * pxPerUnit * h²`。我代入算了一下：`G = 9.81`、`pxPerUnit ≈ 60`（取决于视口大小）、`h = 1/240 s`。结果是每个子步长只能产生 **0.022 像素**的位移。4 个子步长加起来，每帧才移动大约 0.09 像素。屏幕上一根头发丝的宽度都不到，肉眼根本看不到。

根本原因是 Verlet 和基于角度的物理对重力的处理方式完全不同。在 RK4 里，重力直接产生角加速度，乘以杆长就变成了可观的线加速度。但在 Verlet 里，重力直接作用于像素坐标——`pxPerUnit` 虽然能把物理单位映射到像素，但数值太小了，加上时间步长的平方（`h² ≈ 1.74e-5`），重力就变成了一个几乎可以忽略的量。

**调试过程**：我在浏览器控制台手动输出了几帧的 `y` 坐标变化，发现每帧的 Δy 只有 0.09 像素。然后我在 `verletStep()` 里插了一个 `console.log` 打印 `G * pxPerUnit * dt * dt` 的值，看到 `9.81 * 60 * (1/240)² ≈ 0.01` 的时候才确认问题出在重力量级上。

**怎么修的**：我在重力计算上加了一个乘数 `VERLET_G_SCALE = 8`。乘上去之后，每帧位移 ~0.7 像素，摆终于能正常摆动了。

但说实话，这只是个权宜之计。`VERLET_G_SCALE` 没有物理意义——它是一个纯粹的调参魔术数字。正确的做法应该是用拉格朗日力学在物理单位空间中计算，而不是在像素空间里硬凑。这也为后来第八阶段的重写埋下了伏笔。

**学到什么**：Verlet 积分在粒子模拟中表现良好，特别是在力和约束力数量级相当的时候。但对于重力驱动的摆系统，像素空间和物理空间之间的比例失配会导致严重的问题。`pxPerUnit * h²` 这个乘积太小了，重力项几乎消失了。如果在 Verlet 里遇到重力看起来不工作的情况，先检查这个乘积的数值量级。

## 2. 关节修改后轨迹不同步

**背景**：做完 Verlet 基础实现之后，我开始调试关节的添加和移除功能（`addJoint` / `removeJoint`）。每个粒子的轨迹存储在 `trails[i]` 数组中。

**有多疼**：低。视觉闪烁。

**发生了什么**：点了 `➕` 按钮添加关节之后，控制台报错了。错误信息指向 `drawTrail` 函数，说是读取 `trails[i]` 时 `i` 越界了。我数了一下——粒子有 N 个，但轨迹数组只有 N-1 个。

**为什么**：`addJoint` 函数的实现只关注了物理数据：它往 `particles[]` 里 push 了一个新粒子，往 `constraints[]` 里 push 了一个新约束。但 `trails[]` 这个数组被忘记了。添加前有 3 个粒子、3 个约束、3 个轨迹数组；添加后有 4 个粒子、4 个约束，但**仍然是 3 个轨迹数组**。下一帧渲染时，`drawTrail` 遍历第四粒子的轨迹——数组不存在。

同理，`removeJoint` 移除了粒子和约束，但没有移除对应的轨迹数组，导致残留了一个已经不存在粒子的轨迹数据。

**怎么修的**：在 `addJoint` 里为新粒子 push 一个空轨迹数组 `p.trails.push([])`，在 `removeJoint` 里 pop 最后一个 `p.trails.pop()`。

```js
// addJoint 里加一行
p.particles.push({ x, y });
p.constraints.push({ a: p.particles.length - 1, b: p.particles.length - 2, len });
p.trails.push([]);  // ← 这行原来没有

// removeJoint 里加一行
p.particles.pop();
p.constraints.pop();
p.trails.pop();     // ← 这行原来没有
```

另外在 `resizeCanvas` 里也加了一个全量轨迹清除的逻辑，因为窗口大小变化后所有像素位置都变了，旧的轨迹点失去了意义。

**学到什么**：并行数组是反模式。`particles[]`、`constraints[]`、`trails[]` 三个数组本质上应该是一一对应的——它们应该被封装成单个 `Particle` 对象。因为选择了并行数组，每次调整大小都必须原子地更新**所有**数组，漏掉一个就出 bug。这个项目后来为这个设计选择付出了好几次代价。

## 3. Verlet 数值阻尼——混沌消失了

**背景**：Verlet 物理引擎基本跑起来了，摆能动了，N 节链可以了，关节增删没问题了。但有一个根本问题：用户反馈说轨迹看起来"太规律了，缺乏混沌运动，即使初始大角度也一样。"用户问我代码里是不是有小角度近似或者阻尼太大。

**有多疼**：致命。双摆的核心价值就是展示混沌。如果不能混沌，这个项目就失去了意义。

**发生了什么**：我在初始角度设了 135°，关闭了阻尼，等了很久——两个摆锤确实在动，画出来的轨迹也很漂亮，但它们就是不会像真正的双摆那样岔开。轨迹始终是规则的、可预测的、周期性的。

**为什么**：三个问题叠加在一起，每个都在偷能量：

1. **每帧 40 次约束修正**：Verlet 的核心是约束求解器。每帧 4 个子步长，每个子步长跑 10 次约束迭代。每次迭代都把粒子的位置往"正确"方向拉 50%。这意味着每帧要做 4 × 10 = 40 次位置修正。每一次修正都在消耗动能——因为修正的本质是把粒子从"它本应去的地方"拉回到"约束说它可以待的地方"。40 次修正，能量就消耗了 40 次。

2. **没有非线性耦合项**：Verlet 把每个粒子当作独立的质点来处理——每帧先独立施加重力，然后通过约束求解器修正位置。但双摆的混沌来自于两臂之间的耦合：`ω₁² sin(θ₁-θ₂)` 和 `ω₂² sin(θ₂-θ₁)` 这些科里奥利类项。Verlet 的粒子模型根本没有这些项的概念。没有耦合，就没有对初始条件的敏感依赖性——而敏感依赖性恰恰是混沌的发动机。

3. **50/50 分割对内球的"双重摩擦"**：约束求解器在修正位置时，把偏移量 50% 分配给约束的两个端点。对于链式摆，每个内球同时是两个约束的端点——它跟上一级有一个约束，跟下一级也有一个约束。所以每轮迭代，内球要从上一级吸收一半修正，再从下一级吸收一半修正。实际效果是内球的"摩擦"翻倍了。这让数值阻尼更加严重。

这三个问题同时作用的结果是：摆看起来像一个阻尼振荡器，能量慢慢耗散，最终陷入准周期性运动。这不是真正的双摆——这只是两根用约束连在一起的杆。

**调试过程**：我一开始不觉得是 Verlet 的问题，毕竟这是游戏物理里常用的方法。我先检查了是否有代码 bug——重力方向对不对、约束迭代次数够不够、dt 大小合不合理。然后我尝试了 `CONSTRAINT_ITERS = 1` 和 `CONSTRAINT_ITERS = 100`——1 次迭代能量损失小但杆的弹性太大，100 次迭代杆很硬但几乎不摆动。然后我意识到问题出在方法本身，而不是具体参数。

我在控制台输出每帧的动能和势能，发现总能量从初始约 30 J 开始，在 5 秒内降到了 10 J 以下——能量根本不可能守恒。

**怎么修的**：我完全放弃了 Verlet 积分和约束求解器，用完整的拉格朗日力学 + RK4 积分重写了整个物理引擎。新的求解器计算包含所有三角耦合项的 N×N 质量矩阵，通过 Cholesky 分解求解。这是项目历史上最大的一次重写——基本上替换了 script.js 中所有跟物理相关的代码。

拉格朗日力学的关键优势在于：运动方程是直接从能量函数（拉格朗日量）推导出来的，没有约束修正、没有位置调整、没有 50/50 分割。RK4 积分器保持四阶精度，能量守恒是设计的一部分，而不是碰巧的结果。切换之后，能量漂移率从几十个百分点降到了 0.0000%。

**学到什么**：基于约束的物理（Verlet、PBD）引入了数学上不透明的隐式阻尼。你很难量化每帧损失了多少能量——因为这个阻尼不是一个可以调节的参数，它是约束求解器工作方式的副产品。拉格朗日力学在数学上要重得多（需要解线性系统），但它在设计上就保证了能量守恒。对于展示混沌的应用来说，这是一个生死攸关的取舍——而不是性能或便利性的权衡。

## 4. N 节锁定——刚体 Bug

**背景**：第八阶段用拉格朗日力学重写物理引擎之后，我需要处理 N 节链的情况。最初，我实现了一个 2×2 的拉格朗日求解器（只支持双摆），对于 N > 2 的情况，我在杆 2 的方向上按比例放置中间粒子。

**有多疼**：致命。N > 2 的摆在物理上就是错的。

**发生了什么**：我打开页面，添加了一个关节，变成三节摆。摆锤 2 和摆锤 3 **完全平行运动**。不管我怎么调初始角度，摆锤 3 永远跟摆锤 2 保持同一方向，关节处没有任何独立弯曲。用户报告说："最后两个摆锤完全锁定在一起，像一个刚体一样运动。"

**为什么**：问题出在两个地方：

首先，`computeParticlePositions` 的实现方式是：从固定点出发，沿 θ₁ 方向放摆锤 1，从摆锤 1 沿 θ₂ 方向放摆锤 2，从摆锤 2 沿 θ₂ 方向放摆锤 3。摆锤 3 没有自己的角度——它沿用了摆锤 2 的 θ₂。所以三个粒子实际上只有 2 个自由度。

其次，物理求解器是硬编码的 2×2 系统：
```
2 l₁ α₁  +  l₂ cos(Δ) α₂  =  -2g sin θ₁  -  l₂ ω₂² sin(Δ)
 l₁ cos(Δ) α₁  +  l₂ α₂  =  -g sin θ₂   +  l₁ ω₁² sin(Δ)
```
这里只有 θ₁ 和 θ₂ 两个变量。当 N > 2 时，θ₃、θ₄……都没有对应的方程。所以即使用了拉格朗日力学，自由度数还是 2。

这两个问题我都犯了：几何上沿用了上一级的角度，物理上也只有 2 个自由度。摆锤 3 的"关节"只是一个几何上的标记，没有任何物理意义。

**调试过程**：我一开始怀疑是渲染问题——也许 `computeParticlePositions` 算对了位置，但 `drawTrail` 画错了？我仔细检查了每个粒子的 `x, y` 值，发现它们的相对方向确实是恒定的。然后我确认了问题出在物理模型：摆锤 3 根本没有自己的 θ₃。

**怎么修的**：我把硬编码的 2×2 求解器替换成了通用的 N 摆拉格朗日求解器。新求解器构建 N×N 质量矩阵：

```
Mᵢⱼ = lᵢ lⱼ (N−max(i,j)) cos(θᵢ−θⱼ)
bᵢ  = −Σⱼ≠ᵢ Aᵢⱼ sin(θᵢ−θⱼ) ωⱼ² − g lᵢ (N−i) sin(θᵢ)
```

这个矩阵适用于任意 N。每个连接现在在 `thetas[]` 数组里有自己独立的角度。`computeParticlePositions` 的逻辑也改成了前向遍历：从固定点开始，每个粒子从上一个粒子的位置按自己的角度放置。不再有任何"沿杆 2 方向插值"的 hack。

**学到什么**：当从 N=2 推广到 N>2 时，架构必须一开始就是 N 参数化的。事后补自由度——"沿着杆 2 插值就行"、"先跑 2×2 求解器再分配角度"——会产生几乎不可能仅凭肉眼观察诊断的耦合伪影。正确的做法是：从一开始就让状态数组支持 N 个独立的角度，求解器也处理 N×N 矩阵。这确实多写了几十行代码，但避免了后续无穷无尽的调试。

## 5. 右键点击让上下文菜单消失

**背景**：第八阶段做完了 N 节链的基础功能之后，开始处理各种交互细节。上下文菜单在选中摆时显示，提供颜色切换、可见性、关节增减、删除等操作。

**有多疼**：中。UI 行为让人困惑。

**发生了什么**：我测试的时候，习惯性地在画布上右键了一下——上下文菜单消失了。我觉得可能是正常的（点击空白区域取消选中），但用户报告说："在屏幕上右键点击时，底部的控制图标（调色板、加号按钮等）都消失了。"

**为什么**：`canvasB` 的 `mousedown` 事件处理程序没有检查 `e.button`。在 JavaScript 的鼠标事件中，`click`、`mousedown`、`mouseup` 对所有鼠标按钮都会触发，包括右键和中键。我的处理程序在空白处点击时调用了 `selectPendulum(null)`——不管点的是左键还是右键。右键点击（`e.button === 2`）也一样触发了取消选择，隐藏了上下文菜单。

这个问题在开发过程中从来没有暴露过，因为我几乎不用右键点击 Canvas。而且即使右键点了，上下文菜单消失的瞬间我也没有太在意——直到用户明确报告这是一个 bug。

**怎么修的**：在 `canvasB` 的 `mousedown` 处理程序开头加了一行守卫：

```js
if (e.button !== 0) return;  // 只有左键触发选择/取消选择
```

同时还加了 `canvasB.addEventListener('contextmenu', (e) => e.preventDefault())` 来阻止浏览器原生的右键菜单弹出——这样即使用户右键点击了摆锤，也不会弹出浏览器的上下文菜单干扰体验。

**学到什么**：鼠标事件会对所有按钮触发，除非你显式过滤。`event.button` 的值是：0 = 左键、1 = 中键（滚轮）、2 = 右键。在 Canvas 应用中，几乎所有交互都应该只响应左键。这个规则很简单，但很容易忘记加——因为开发者在测试的时候都是潜意识地用左键操作，永远不会触发右键的问题。把 `e.button !== 0` 守卫作为 Canvas 事件处理的标准模板，可以在 Source 阶段就避免这类问题。

## 6. 能量不守恒——势能公式符号错了

**背景**：做完物理引擎之后，我开始实现实时能量分析功能（Phase 8）。`computeEnergy()` 从拉格朗日量计算动能和势能，`renderEnergyPlot()` 用 Canvas 画出三条线：KE（动能）、PE（势能）、totalE（总能）。我的验收标准很明确：总能量必须守恒。

**有多疼**：高。341% 的能量漂移让能量图毫无意义。

**发生了什么**：能量图画出来了，三条线很漂亮——除了总能量在 -29 J 到 +113 J 之间剧烈振荡。一个守恒量理论上应该是一条水平的直线，但它看起来像地震仪记录。我算了一下漂移：341%。也就是说能量既会"凭空产生"又会"凭空消失"——这在物理上显然是不可能的。

**为什么**：我对势能的定义是：

```
PE = g * Σᵢ (N−i) * lᵢ * cos(θᵢ)
```

这个公式在画布坐标系（y 轴向下）中是直觉的：重力向下，位置越低的摆锤势能越小。但问题是拉格朗日方程使用的是物理坐标系（y 轴向上）。

在运动方程中，重力项的符号是 `−g (N−i) lᵢ sin(θᵢ)`。这个符号来自拉格朗日量 `L = T − V`，其中势能 V 的符号决定了力的方向。要让运动方程中的重力项符号正确，势能必须是：

```
V = −g * Σᵢ (N−i) * lᵢ * cos(θᵢ)
```

跟我的公式**差了一个负号**。所以总能量 `totalE = KE + PE` 中，PE 的符号错了——当摆锤向下运动时，PE 应该减小（传递给了 KE），但因为我用了正号，PE 在增大，导致总能量看起来像在做布朗运动。

**调试过程**：我花了很长时间检查 `computeEnergy` 的实现，核对了拉格朗日量的每个项，都没有语法错误。我一度怀疑是 `computeAccel` 或者 `rk4Step` 有能量漂移问题。输出每一步的 KE 和 PE 之后，我注意到 KE 和 PE 的相位总是相反的——PE 大的时候 KE 也大，PE 小的时候 KE 也小。这在物理上是荒谬的：势能转化成动能，此消彼长才对。然后我意识到：符号反了。

**怎么修的**：一行代码：

```js
// 之前（有问题）：
PE = g * sum;

// 之后（正确）：
PE = -g * sum;
```

`g` 改成 `-g`，一个字符。改完用了两秒钟，验证几步之后能量漂移从 341% 降到了 **0.0000%**。

**学到什么**：坐标系约定在物理模拟中极其重要。画布的 y 轴（向下为正）和物理学的惯例（向上为正）是相反的。每个涉及垂直方向的公式都必须审计符号一致性。这类错误极其隐蔽——代码本身没有 bug，公式看起来也对，但就是不工作的原因是"符号约定不匹配"。拉格朗日力学尤其容易踩这个坑，因为势能 V 在拉格朗日量 `L = T − V` 中的符号决定了运动方程的所有符号。

## 7. 空白图表——索引没传过去

**背景**：在实现 Phase 8 的指标图表渲染函数时，我设计了一个通用的渲染管线：`drawSolidLine` 接受一个数据数组和两个 getter 函数（`getX` 和 `getY`），遍历数组绘制一条连续的线。

**有多疼**：中。两个图表都是空白。

**发生了什么**：能量图和相空间图都是空白的。没有报错、没有崩溃、没有异常——Canvas 就是干净的黑色背景。我以为是数据收集的问题，但控制台输出 `metricsData` 显示数据是正常的。然后我以为是缩放问题——也许点在 canvas 外面？但也不是。

**为什么**：`drawSolidLine` 的实现中有一行：

```js
const px = mapX(getX(data[i]), xMin, xMax, PLOT_W);
```

而 `getX` 函数的定义是：

```js
(data, i) => i  // 能量图的 X 是数组索引
```

问题在于：`getX(data[i])` 只传了一个参数（当前数据点），但 getter 期望的是 `(value, index)`——它需要第二个参数来返回索引。没有传的话，`i` 在 getter 内部是 `undefined`，返回的是 `undefined`。

然后 `mapX(undefined, xMin, xMax, PLOT_W)` 产生 `NaN`。所有的 X 坐标都是 `NaN`。Canvas 的 `lineTo` 对 `NaN` 参数静默忽略——路径存在但不绘制任何东西。所以画布是空的。

有趣的是，相图的 getter `d => d.thetas[0]` 只使用第一个参数，所以它**不会**触发这个 bug。能量图的 getter `(_, i) => i` 需要第二个参数，所以**会**触发。但我的测试总是先看相图——相图有数据，我以为能量图的数据有问题，但实际上数据是好的，只是 X 坐标是 NaN。

**怎么修的**：

```js
// 之前
const px = mapX(getX(data[i]), xMin, xMax, PLOT_W);

// 之后
const px = mapX(getX(data[i], i), xMin, xMax, PLOT_W);
```

显式传入了第二个参数 `i`，getter 就能正确返回索引了。

**学到什么**：JavaScript 函数可以接受任意数量的参数——多余的被忽略，缺少的是 `undefined`。这种灵活性在日常使用中很方便，但当参数通过工厂函数传递时，会导致很难追踪的静默错误。如果 API 的签名是 `(value, index)`，那调用方必须始终传两个参数。TypeScript 或者 JSDoc 可以避免这类问题，但在纯 JS 项目中，只能靠编码时的细心。

## 8. 多摆指标——全局数组 Bug

**背景**：做完能量图的基础实现之后，我开始处理多摆的场景。用户可以添加多个独立摆，每个摆的指标应该独立跟踪。

**有多疼**：高。切换摆的时候相图和能量图显示错误数据。

**发生了什么**：页面上有两个摆。我选中摆 A，相图显示的是摆 A 的 θ₁ 和 ω₁——看起来没问题。然后我点击摆 B，切换选择——**相图没有变化**。还是显示摆 A 的数据。我重新选中摆 A，摆 B 的数据也丢了。

**为什么**：我用了一个**全局**的 `metricsData` 数组：

```js
let metricsData = [];
function collectMetrics() {
    const p = selectedPendulum ?? pendulums[0];
    if (!p || !p.visible) return;
    metricsData.push({ thetas: [...p.thetas], omegas: [...p.omegas], KE, PE, totalE });
    while (metricsData.length > METRICS_CAPACITY) metricsData.shift();
}
```

问题很明显了：`collectMetrics` 每帧只往一个全局数组里 push 数据，这个数据属于"当前选中的那个摆"。当用户切换选择时，全局数组里混入了两个摆的数据——没有区分它们属于谁。而图表渲染器 `renderPhasePortrait` 读的是同一个全局数组，不知道数据属于哪个摆。

更隐蔽的问题是：当用户新生成一个摆时，它还没有被选中的机会，`collectMetrics` 不会为它收集任何数据。渲染器面对这个摆的时候，没有任何可以显示的数据。

**怎么修的**：每个摆现在拥有自己的 `metrics[]` 数组，在 `createPendulum()` 中初始化：

```js
function collectMetrics() {
    for (const p of pendulums) {
        if (!p.visible) continue;
        p.metrics.push({ thetas: [...p.thetas], omegas: [...p.omegas], KE, PE, totalE });
        while (p.metrics.length > METRICS_CAPACITY) p.metrics.shift();
    }
}
```

图表渲染器通过 `getTrackedPendulum()` 获取当前应显示的摆，然后读取它的 `p.metrics` 数组。切换选择时，`getTrackedPendulum()` 返回新的摆，渲染器自动使用新数据。删除摆时，它的 metrics 数组随对象一起被垃圾回收。

**学到什么**：当数据是每个实体独立的时候，存储也应该是每个实体独立的。全局缓冲区在收集和渲染之间创建了隐藏的耦合，这种耦合只在特定的交互序列下才会暴露——例如多摆场景下的选择切换。如果一开始就把 `metrics` 设计为每个摆的成员变量，这个 bug 从一开始就不存在。这是"数据亲和性"的一个典型案例：数据应该附着在它描述的对象上。

## 9. 能量图 X 轴停滞

**背景**：指标数据收集正常工作之后，我开始观察能量图的行为。每个摆的 metrics 数组是一个 300 点的环形缓冲区——存满后新数据覆盖旧数据。

**有多疼**：中。图表看起来冻住了。

**发生了什么**：打开能量图，一开始 X 轴从 0 到 50、到 100、到 200……稳步增长。但到第 300 帧之后，X 轴的刻度标签停了。永远显示 0、60、120、180、240、299。图上三条线确实在动——它们在横向压缩，因为新数据还在进来，但 X 轴范围不变。

**为什么**：能量图 X 轴的取值逻辑是 `(_, i) => i`——数组索引。而 `xRange.max = data.length - 1`。当 metrics 数组长度从 0 增长到 300 的过程中，X 轴是正确增长的。但一旦数组满了（`data.length === 300` 保持不变），`data.length - 1` 就永远等于 299。

根本原因：环形缓冲区的长度是固定的——数组满了之后，`shift()` 和 `push()` 的操作保持长度不变。所以用 `data.length` 作为 X 轴范围就永远停滞了。

**怎么修的**：我引入了一个全局计数器 `globalMetricsStep`，每物理帧递增。每个指标条目在收集时保存当前的步数：

```js
p.metrics.push({
    thetas: [...p.thetas],
    omegas: [...p.omegas],
    KE, PE, totalE,
    step: globalMetricsStep  // ← 新增
});
globalMetricsStep++;
```

能量图现在使用 `d => d.step` 作为 X 轴 getter，范围来自第一个和最后一个条目的 `step`：

```js
const first = data[0];
const last = data[data.length - 1];
const xRange = { min: first.step, max: last.step };
```

由于环形缓冲区里始终是 `[step_N-299, ..., step_N]`，X 轴范围每帧向前滑动 1。刻度标签持续更新，用户可以看到时间在流逝。

指标清除时（关掉再打开指标面板，或者重置模拟），`globalMetricsStep` 也重置为 0。

**学到什么**：环形缓冲区把数据数量和时间进程解耦了。`data.length` 在环形缓冲区中不是时间的代理——它只是一个固定上限。画滚动窗口时序图的时候，X 轴必须用一个外部单调递增的序列号，而不是数组索引。任何使用固定长度缓冲区的可视化都需要注意这一点。

## 10. 相图环绕线伪影

**背景**：相空间图是 θ₁（X 轴）对 ω₁（Y 轴）的散点轨迹。为了让图稳定可读，我把所有 θ₁ 值归一化到 [0°, 360°) 范围——这样角度不会因为多次旋转而累加到 1080°。

**有多疼**：中。相图上出现误导性的线条。

**发生了什么**：摆锤旋转很多圈之后，相图上出现了一条**笔直的水平线**，从 X 轴最右端连到最左端，横穿整个吸引子轨迹。看起来像是某种故障，但实际上是一个数学边界问题。

**为什么**：当 θ₁ 从约 359° 绕回到约 1° 时，它跨越了 0°/360° 的边界。归一化后的数据中，前一个点是 359°，后一个点是 1°——它们在数组中相邻。`drawFadingLine` 的路径绘制逻辑只是简单地从前一个点的位置 `lineTo` 到后一个点的位置——它不知道这两个点在角度空间中是相邻的（只差 2°），在数值空间中却相隔了 358°。所以它在图上画了一条 358° 宽的直线。

**怎么修的**：在 `drawFadingLine` 中，每次连接两个连续数据点之前，检查 X 值的跳跃是否超过数据范围的一半：

```js
const wrapThreshold = range * 0.5;
if (Math.abs(x2 - x1) > wrapThreshold) {
    // 检测到环绕：描边当前路径，开始新路径
    ctx.stroke(path);
    path = new Path2D();
    path.moveTo(currentX, currentY);
}
```

阈值设为数据范围的一半（即 π，因为 X 范围是 [0, 2π)）。如果两个连续点的 X 差超过 π，就认为发生了角度环绕，断开路径连接。这样从 359° 到 1° 的"跳跃"不会在图上画出那条错误的线。

由于路径在环绕点断开，这条线就消失了。轨迹环绕点变成了一条开放路径的末端和另一条开放路径的开端——在视觉上几乎不可察觉。

**学到什么**：角度数据有线性数据没有的拓扑特性——它是一个圆而不是一个区间。任何绘制角度量的可视化都必须处理周期性边界条件。`0° ≈ 360°` 这个在生活中显而易见的常识，放在路径绘制中就变成了一个需要显式处理的边界问题。数据上的相邻不等于视觉上的连接。

## 11. 角度显示点击被拦截

**背景**：Phase 8 的迭代中，我把左上角的角度显示从静态文本改成了可点击的摆选择器。每个摆渲染为一个带 `data-idx` 属性的 `<div>`，点击可以切换选中的摆。

**有多疼**：中。摆选择器时灵时不灵。

**发生了什么**：在指标面板关闭的情况下，点击角度显示中的摆条目可以正常切换选择。但一旦指标面板打开，或者有图表被放大了，点击就不反应了。最令人困惑的是——没有报错、没有闪烁、没有任何反馈，就像点击事件根本没有发生过。

**为什么**：两个问题叠加：

1. 图表画布的 `click` 处理程序中调用了 `e.stopPropagation()`。`stopPropagation()` 阻止事件冒泡到父元素。角度显示的 `click` 监听器是使用 `document.getElementById('angle-display').addEventListener('click', ...)` 注册的——它依赖事件冒泡来捕获点击。当图表上的 `click` 事件被 `stopPropagation()` 拦截后，事件到不了 `angle-display` 的监听器。

2. 更根本的问题：当我点击放大某个图表时，它占满了 90% × 88% 的视口，`z-index: 10`，并且在 CSS 上位于角度显示的上面。指针事件在 DOM 树中通常先穿透到 z-index 高的元素——所以点击事件被图表捕获了，然后就被 `stopPropagation()` 拦住了。

**调试过程**：我花了一些时间才确定问题是 `stopPropagation` 而不是事件丢失——因为我试了在 `angle-display` 上添加 `console.log` 发现根本没触发。然后我检查了是否有元素遮挡——确实有，但即使没遮挡也没触发。最后我意识到是 `click` 事件传播的问题。

**怎么修的**：我把事件监听从 `click` 换成了 `pointerdown`，使用文档级事件委托：

```js
document.addEventListener('pointerdown', (e) => {
    const entry = e.target.closest('.pend-entry');
    if (!entry || !document.getElementById('angle-display').contains(entry)) return;
    // 选择摆...
});
```

`pointerdown` 是 `Pointer Events API` 的一部分，它在触摸或鼠标按下时**立即**触发，不受 `click` 处理程序的 `stopPropagation()` 影响。而且文档级别的监听器优先于任何元素的监听器——无论 z-index 如何，`pointerdown` 总会触发。

**学到什么**：`click` 是一个组合事件——它在 `mousedown` + `mouseup` 在同一元素上发生时合成。任何一个阶段如果被 `stopPropagation()` 阻止，`click` 就不会触发。`pointerdown` 则是在按下时立即触发，不依赖事件合成。对于需要在覆盖层或弹窗下依然保持可交互的 UI，使用 `pointerdown` 比 `click` 更可靠。这不是什么黑魔法——只是事件模型的不同层级。

## 12. 切换选择时能量图标题不更新

**背景**：可点击角度显示正常工作之后，我发现相图标题始终显示"摆 1"，即使已经切换到了摆 2。

**有多疼**：低。装饰性问题，但让人困惑。

**发生了什么**：打开指标面板，显示的是摆 A 的相图，标题写着"Pendulum 1 θ₁ vs ω₁"。然后我在角度显示中点击摆 B——图表更新了，显示摆 B 的数据。但标题**仍然是**"Pendulum 1"。

**为什么**：`renderPhasePortrait()` 的函数结构是：

```js
function renderPhasePortrait() {
    const p = getTrackedPendulum();

    // 更新标题（这里其实没问题）
    updateTitle(p.id);

    // 数据守卫——数据不够就提前返回
    if (p.metrics.length < 2) return;

    // 绘制图表...
}
```

我检查代码的时候发现标题更新确实在函数开头——那为什么没更新呢？

其实真正出问题的是**另一个版本**的代码，在多次迭代中函数结构发生了变化。bug 版本的结构是：

```js
function renderPhasePortrait() {
    const p = getTrackedPendulum();

    // 数据守卫——数据不够就提前返回
    if (p.metrics.length < 2) return;

    // 更新标题
    updateTitle(p.id);

    // 绘制图表...
}
```

当选择切换到摆 B 时：
- `getTrackedPendulum()` 返回摆 B
- 摆 B 刚被创建，metrics 数组为空或只有 1 个条目
- `p.metrics.length < 2` → **提前返回**
- 标题更新**永远不会执行**
- 下一帧：摆 B 有了新数据，但标题在数据守卫**之后**才更新——已经在上一帧被跳过了

由于 Canvas 在面板关闭时不重绘，重新打开面板时调用 `renderPhasePortrait()` 再次被 `metrics.length < 2` 挡住。直到很多帧之后，摆 B 的数据足够多了，标题才会更新。

**怎么修的**：把标题更新移到数据守卫之前：

```js
function renderPhasePortrait() {
    const p = getTrackedPendulum();

    // 先更新标题——不需要数据也知道标题是什么
    updateTitle(p.id);

    // 再检查数据
    if (p.metrics.length < 2) return;

    // 绘制图表...
}
```

**学到什么**：函数中提前返回的守卫不应该阻止元数据（比如标题、标签、轴范围）的更新。即使数据不可用，标题也可以正确显示当前选中了哪个摆。操作顺序很重要：非数据依赖的更新应该放在数据依赖的操作之前。

## 13. CSS 优先级战争——代码块渲染错乱

**背景**：我实现了应用内 Wiki 阅读器——点击指南弹窗中的"查看完整指南"会在应用内加载和渲染 README。README 中包含围栏代码块（使用 ``` 标记的 ASCII 目录树）。

**有多疼**：中。文档显示坏了。

**发生了什么**：指南弹窗里的代码块全部显示为单行。README 的架构目录树：

```
Double-Pendulum-Visualizer/
├── index.html
├── style.css
├── script.js
```

变成了：

```
Double-Pendulum-Visualizer/ ├── index.html ├── style.css ├── script.js
```

代码块内的换行符被完全忽略了。

**为什么**：`style.css` 中有一条规则：

```css
#help-modal-content code {
    white-space: nowrap;
}
```

这条规则的优先级是 1,0,0,1（一个 ID + 一个元素）。它应用于弹窗内**所有** `<code>` 元素，包括 `<pre>` 块内部的代码——而 `<pre>` 块应该保留空白字符。

我的修复规则是：

```css
.help-readme-body pre code {
    white-space: pre;
}
```

优先级是 0,2,0,2（两个类 + 两个元素）。在 CSS 优先级系统中，ID 选择器（1,0,0,0）比类选择器（0,1,0,0）高。所以无论我的规则在样式表中的位置如何，`#help-modal-content code` 始终胜出。

我试过把规则移到样式表的最后、移到最前、甚至单独放在一个全新的文件中——**都完全没用**。因为 CSS 优先级是权重系统，不是顺序系统。

**怎么修的**：在 `pre code` 选择器中也包含父级 ID，使优先级更高：

```css
#help-modal-content .help-readme-body pre code {
    white-space: pre;
}
```

`#help-modal-content .help-readme-body pre code` 的优先级是 1,1,2,1——它胜过了 `#help-modal-content code` 的 1,0,0,1。

**学到什么**：CSS 优先级是一个权重系统，不是顺序系统。`#id`（权重 0-1-0-0）永远高于 `.class`（权重 0-0-1-0）。重排 CSS 规则的位置是完全没有意义的——唯一的解决办法是增加选择器的权重。常见做法有两种：在子选择器中加入父级 ID，或者（更不推荐）使用 `!important`。后者会增加技术债务。

这个 bug 花了我很长时间，不是因为技术复杂——而是因为我花了几个小时在样式表里移来移去，期待 CSS 表现得像"后面覆盖前面"一样。但实际上，当规则有不同的选择器权重时，顺序根本不重要。

## 14. 移动端触控弹跳（iOS Safari）

**背景**：第九阶段的目标是让应用在手机上可用。核心交互——拖拽摆锤设置角度——在触摸设备上必须工作。

**有多疼**：高。iOS 上根本没法拖拽。

**发生了什么**：我在 iPhone 上打开页面，暂停模拟，点击一个摆锤——页面弹性滚动，摆锤不动。不管我怎么划，都是页面在动，摆锤纹丝不动。

**为什么**：iOS Safari 的弹性过度滚动（rubber-band scrolling）劫持了所有的触摸事件。就算在 `<body>` 上设置了 `overflow: hidden`，Safari 仍然会拦截触摸事件来做页面滚动。这不是一个 bug——这是 Safari 的设计行为，因为用户预期在任何页面上都能滚动。

更隐蔽的问题是：我原来的触摸事件监听器用的是 `{ passive: true }`。在 `passive: true` 的情况下，浏览器**忽略** `preventDefault()` 的调用——因为 `passive` 的意思是"我保证不会调用 preventDefault，你可以安全地优化"。所以即使用了 `preventDefault()`，Safari 还是能自由地处理触摸滚动。

**调试过程**：我一开始在 `touchstart` 里加了 `e.preventDefault()` 但没有效果。查了 MDN 之后才发现 `passive: true` 的语义——浏览器会忽略 preventDefault。这不是 iOS 特有的行为——所有现代浏览器都支持 `passive`，但 iOS Safari 是唯一默认启用了弹性滚动的。

**怎么修的**（三层防御）：

第一层：CSS 级别的阻断：

```css
canvas {
    touch-action: none;
}
```

`touch-action: none` 在合成器级别告诉浏览器永远不要把这个元素上的触摸手势解释为滚动或缩放。这层防御最可靠，因为它在 GPU 合成线程上工作，不依赖 JavaScript 的执行。

第二层：文档级别的弹跳阻断：

```css
html, body {
    overscroll-behavior: none;
}
```

防止在文档边界处的弹性弹跳效果。

第三层：JavaScript 级别的精确控制：

```js
// 从 { passive: true } 改为 { passive: false }
canvasB.addEventListener('touchstart', handleTouchStart, { passive: false });
canvasB.addEventListener('touchmove', handleTouchMove, { passive: false });

function handleTouchStart(e) {
    // ... 检测是否点击到了摆锤 ...
    if (命中摆锤) {
        e.preventDefault();  // 现在不会被忽略了
        dragActive = true;
    }
}
```

注意 `preventDefault()` 只在拖拽激活时有条件地调用——不干扰非拖拽的触摸操作。

**学到什么**：移动端触摸处理需要纵深防御，单靠一层是不够的。CSS 级别的 `touch-action: none` 是最可靠的（在合成器线程工作），但仍然需要 JS 级别的 `passive: false` + `preventDefault()` 来完全控制。这三层不是冗余的，它们是互补的——每一层处理不同线程和不同层级的触摸拦截。

## 15. 启动时画布空白

**背景**：在第九阶段的迭代中，我移除了 slowMo 模式相关的代码（`slowMo` 变量和 `btn-slow` 按钮）。这本来是一个简单的清理操作。

**有多疼**：高。启动时摆看不见。

**发生了什么**：清理代码之后，我刷新页面——只有暗色背景，摆消失了。没有报错、没有崩溃、没有控制台消息——就是看不见摆。我以为是物理引擎崩了，但打开控制台看到状态数据是正常的。

**为什么**：两个独立的问题：

1. **缺少同步初始绘制**：应用的引导流程是 `addPendulum()` → `resizeCanvas()` → `updateControls()` → `animate()`。而 `animate()` 通过 `requestAnimationFrame(animate)` 排队第一帧——这大约在 16ms 后、下一个 VSync 信号到达时才会触发。在这 16ms 的间隙里，画布是透明的（从未被绘制过）。

这个间隙在正常情况下是无关紧要的——16ms 后动画开始，第一帧绘制所有内容。但在某些浏览器或某些条件下（比如 DevTools 打开、CPU 降频），这个间隙可能更长。用户在看到一个空白页面的瞬间可能就会刷新。

2. **不安全的 `matchMedia` 使用**：我的移动端检测代码是：

```js
const isMobile = window.matchMedia("(max-width: 768px)").matches;
```

`window.matchMedia` 在大部分浏览器中可用，但在某些环境（比如无头浏览器、某些 WebView 实现、测试框架）中可能不存在或抛出异常。如果 `matchMedia` 不存在，这条语句会抛出 `TypeError`，导致整个顶层脚本执行停止——`addPendulum`、`resizeCanvas`、`animate`——全部不会执行。

但等等，我移除 slowMo 代码的时候并没有改 `matchMedia` 这行——那为什么之前能用，移除 slowMo 之后不能用了？

实际上，slowMo 的移除**暴露**了这个问题：在 slowMo 存在的版本中，`animate()` 的第一帧会额外执行一些渲染路径（因为 slowMo 按钮的存在改变了某些绘制状态）。移除 slowMo 后，这个"额外渲染"消失了，暴露了同步初始绘制缺失的问题。而 `matchMedia` 的脆弱性一直存在——只不过之前的测试环境恰好有 `matchMedia`。

**调试过程**：我加了几个 `console.log` 来跟踪执行流程——所有初始化代码都正确执行了，`animate` 也被调用了。问题不在执行流程，而在于**时机**。我用 `performance.now()` 在 `animate` 的第一帧打了一个时间戳——确实比页面加载晚了几十毫秒。然后我在 `addPendulum` 之后立即加了一个 `draw()` 调用——摆出现了。

**怎么修的**：
- 添加 `window.matchMedia` 空守卫：`window.matchMedia ? window.matchMedia("...").matches : false`
- 在 `animate()` 之前添加**同步**的 `draw()` 和 `updateAngleDisplay()` 调用

```js
addPendulum();
resizeCanvas();
updateControls();
draw();                     // ← 新增：同步绘制
updateAngleDisplay();       // ← 新增：同步更新角度显示
animate();                  // 现有的 rAF 调用
```

这样在第一个布局帧上就立刻渲染了摆和角度，不等 rAF 回调。

**学到什么**：`requestAnimationFrame` 回调不是即时的——它排队到下一个 VSync，通常 16ms。如果应用依赖用户立即看到某些视觉元素，就需要一个同步的绘制路径。另外，`window.matchMedia` 不是所有 JavaScript 环境都提供的 API——必须加守卫。浏览器 API 的可用性不应该靠假设。

## 16. GitHub Pages 部署崩溃

**背景**：应用在本地完美运行之后，我推送到 GitHub Pages 上部署。这是一个纯前端的静态应用，理论上部署应该是零摩擦的。

**有多疼**：致命。应用在生产环境中完全无法运行。

**发生了什么**：打开 GitHub Pages 链接——只有空白暗色页面。打开控制台，一行错误：

```
TypeError: Cannot read properties of null (reading 'addEventListener')
```

动画循环从未启动。控制按钮从未绑定事件。所有交互全部失效。用户看到的是一个死页面。

**为什么**：两个问题叠加：

1. **缺少 `defer`**：HTML 文件中的 `<script>` 标签：

```html
<script src="script.js" defer></script>
```

如果 `defer` 存在，浏览器会先解析完所有 HTML（构建 DOM 树），再执行脚本。但我的原始版本**没有** `defer`：

```html
<script src="script.js"></script>
```

没有 `defer` 的情况下，浏览器在解析到 `<script>` 标签时**立即暂停 HTML 解析**，下载并执行脚本。在本地开发中（`file://` 或 `localhost`），HTML 文件从本地磁盘读取，毫秒内就完全加载——脚本执行时所有 DOM 元素都已经存在。

但在 GitHub Pages 上，文件通过 CDN 传输，响应可能被分块（chunked transfer encoding）。如果 HTML 的传输被网络延迟拆成了多个数据块，浏览器可能在第一个数据块到达并解析到 `<script>` 标签时就执行脚本——此时后续的 DOM 元素还没有被解析。

2. **ID 不匹配**：脚本中有类似这样的代码：

```js
document.getElementById('some-element-id').addEventListener(...);
```

在本地测试中，这个元素存在——getElementById 返回有效的 DOM 节点。但在 GitHub Pages 上，由于时机问题（或者 HTML 中的 ID 被更新了但 JS 没同步更新），getElementById 返回 `null`。在 `null` 上调 `.addEventListener()` 抛出了 TypeError。

这个错误从顶层脚本传播出去——JavaScript 中未捕获的错误会停止后续所有代码的执行。所以 `animate()` 没有被调用、按钮事件没有被绑定、一切交互失效。

最棘手的是：在本地环境中**永远无法复现这个 bug**。因为本地文件加载是毫秒级的，永远不会出现"HTML 还没加载完脚本就执行了"的情况。

**怎么修的**：
- 给 `<script>` 标签加上 `defer`：确保浏览器在完整解析 HTML 之后才执行脚本
- 创建了 `on(id, event, handler)` 安全绑定函数：

```js
function on(id, event, handler) {
    const el = document.getElementById(id);
    if (!el) {
        // 元素不存在的静默跳过——不抛异常
        return;
    }
    if (handler) {
        el.addEventListener(event, handler);
    }
    return el;
}
```

- 把项目中所有 18 处 `document.getElementById(id).addEventListener(event, handler)` 调用改成了 `on(id, event, handler)`

**学到什么**：本地开发环境会隐藏一切跟时机相关的 bug。`file://` 协议的加载速度掩盖了"先有 DOM 还是先有 JS"的顺序问题。GitHub Pages（以及任何 CDN 部署）通过真正的网络延迟暴露了这些问题。

`defer` 应该是任何引用 DOM 元素的 `<script>` 的默认选择——即使在不使用 `defer` 也不会出问题的本地环境中。安全绑定的辅助函数（`on(id, event, handler)`）也应该作为标准实践——对 `null` 调用方法是 JavaScript 中最常见也最容易被忽略的生产环境崩溃原因之一。

## 17. 极端角度下 Cholesky 分解的正定性问题

**背景**：在做物理求解器优化时，我用 Cholesky 分解替换了高斯消元。Cholesky 要求矩阵是对称正定的（SPD）。在数学上，质量矩阵 M 具有类 Gram 结构，天然是 SPD 的。

**有多疼**：低（被安全网接住了）。但如果安全网不存在，会造成静默数据损坏。

**发生了什么**：理论上，在极端角度和高 N（7-8 节）时，M 可能失去正定性，导致 `Math.sqrt()` 处理对角元时返回 NaN。NaN 会通过求解器传播到角度和角速度——导致所有粒子坐标变成 NaN，轨迹消失。

**为什么**：在数学上，质量矩阵 M 始终是对称正定的——它的结构保证了对于任何非零向量 x，`x^T M x > 0`。但在浮点数表示中，当矩阵接近奇异时（在极端角度组合下），浮点舍入误差可能导致对角线元素变成微小的负数（比如 `-1e-16`）。

`choleskyFactor` 函数中关键的一行是：

```js
L[jN + j] = Math.sqrt(M[jN + j] - diagSum);
```

如果 `M[jN + j] - diagSum` 是负数，`Math.sqrt()` 返回 NaN。然后 NaN 通过前向代入和后向代入传播——求解器返回 NaN 的加速度向量。RK4 步进把这些 NaN 写回到 `thetas[]` 和 `omegas[]`。最终 `computeParticlePositions` 产生 NaN 的 `x, y` 坐标。

在 Canvas 上，用 `NaN` 作为参数的 `arc`、`moveTo`、`lineTo` 被静默忽略——不绘制、不报错。用户看到的就是摆消失了。

**怎么处理的**：我有意**不在求解器级别修复**这个问题。NaN 通过物理状态传播，在一帧后被 `isPendulumInvalid()` / `safeResetPendulum()` 捕获：

```js
function isPendulumInvalid(p) {
    for (let i = 0; i < p.N; i++) {
        if (!isFinite(p.thetas[i]) || !isFinite(p.omegas[i])) return true;
    }
    for (const pt of p.particles) {
        if (!isFinite(pt.x) || !isFinite(pt.y)) return true;
    }
    return false;
}
```

这个函数每帧在 `stepPhysics()` 的末尾被调用。如果检测到 NaN/Infinity，`safeResetPendulum()` 把角度重置到 135°，速度归零，清除轨迹。摆在一帧后以安全状态重新出现。

用户看到的反应：摆突然"跳"到了 135° 的位置，然后继续摆动。看不出是 NaN 导致的崩溃，看起来就像一次"刷新"。

**学到什么**：不是每个数值边缘情况都需要在数学层面修复。有时候系统级的安全网是更便宜、更健壮的解决方案——特别是当触发条件罕见、瞬态且能自我纠正的时候。与其在 Cholesky 分解上加一堆复杂的 fallback 逻辑（切换到高斯消元、调整对角线、重启求解器），不如让 NaN 传播出去，让通用的异常检测器捕获它。

当然，这个策略的前提是安全网确实存在且可靠。如果没有 `isPendulumInvalid` 和 `safeResetPendulum`，这个 bug 会导致静默数据损坏——用户不知道为什么摆会消失。安全网的存在让这个"不修复"的决定变得合理。

## 18. Float64Array 跨变更点同步问题

**背景**：在做最后的性能优化时，我引入了 `p.ls_f64`——一个 `Float64Array` 类型的镜像变量，用于热路径（`computeAccel` 和 `rk4Step`）中读取杆长。`p.ls` 是动态数组（`Array<number>`），`p.ls_f64` 是它的类型化副本。它们需要在每次 `p.ls` 变化时同步。

**有多疼**：低。只在特定时机下才会显现——但在某些情况下会导致静默的物理错误。

**发生了什么**：理论上，如果在某个代码路径中修改了 `p.ls` 但忘记更新 `p.ls_f64`，热路径求解器在下一帧使用 `p.ls_f64` 时会读到**旧**的杆长。摆的链长突然变化（或保持旧值），导致摆动异常。

实际上，这个 bug 并没有在生产中触发——因为我在引入 `p.ls_f64` 时很仔细地标记了每个需要同步的位置。但它是一个**等待发生的 bug**：任何将来新增的修改 `p.ls` 的代码路径都可能忘记同步 `p.ls_f64`。

**为什么**：项目中修改 `p.ls` 的地方有 4 个：
- `addJoint()` — 添加关节后，`p.ls` 长度 +1
- `removeJoint()` — 移除关节后，`p.ls` 长度 -1
- `rebuildChain()` — 重新构建整个链（重置、调整大小时）
- `resizeCanvas()` — 画布大小变化时重建物理

前三个是在代码审查中确定的。第四个 `resizeCanvas` 是在测试中发现的——`pxPerUnit` 的变化改变了所有杆长，但 `p.ls_f64` 没有同步。

**怎么修的**：在所有 4 个修改点后加上 `p.ls_f64.set(p.ls)`：

```js
// addJoint
p.ls = p.constraints.map(c => c.len);
p.N = p.ls.length;
p.ls_f64.set(p.ls);  // ←

// removeJoint
p.ls = p.constraints.map(c => c.len);
p.N = p.ls.length;
p.ls_f64.set(p.ls);  // ←

// rebuildChain
p.ls = chain.constraints.map(c => c.len);
p.N = p.ls.length;
if (p.ls_f64) p.ls_f64.set(p.ls);  // ← if 守卫兼容旧版本创建的摆

// resizeCanvas
p.ls = chain.constraints.map(c => c.len);
p.N = p.ls.length;
p.ls_f64.set(p.ls);  // ←
```

在 `rebuildChain` 中加 `if (p.ls_f64)` 守卫是因为这个函数可能在旧版本的摆上调用（比如从 URL 恢复状态时，某些摆可能是在引入 `ls_f64` 之前创建的）。

**学到什么**：把可变状态镜像到类型数组中会创建一个需要精心维护的同步面。每个修改源数组的代码路径都必须同时更新镜像。更干净的做法是写一个辅助函数：

```js
function syncLsF64(p) {
    if (p.ls_f64) p.ls_f64.set(p.ls);
}
```

然后在所有修改点调用它。这样如果将来再有新的修改点，只需要加一行 `syncLsF64(p)` 就行了。

这个问题的根源是**数据冗余**——同一份信息（杆长）在两个不同的表示中（动态 `Array` 和类型化 `Float64Array`）。冗余本身不是问题，但需要显式的同步策略来维持一致性。
