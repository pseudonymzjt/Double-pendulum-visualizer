# 遇到的困难 — 混沌艺术：双摆可视化器

> 本文档记录了开发过程中遇到的重大工程挑战、其根本原因以及最终采用的解决方案。

---

## 1. Verlet 数值阻尼（第七阶段 → 第八阶段重写）

**严重程度**：致命——导致摆体完全不产生混沌。

**症状**：双摆轨迹"看起来太规律了，缺乏混沌运动，即使在初始大角度下也是如此。"用户特别询问是否存在小角度近似或过大的阻尼。

**根本原因**：第七阶段的 Verlet + 约束求解器方法存在三个叠加问题：

1. **每帧 40 次约束修正**：4 个子步长 × 10 次约束迭代 = 每帧 40 次位置调整，每次应用 50/50 的修正分割。这起到了隐式阻尼的作用——重复的修正每帧都会消耗动能。

2. **无非线性耦合**：Verlet 独立处理每个粒子（重力 + 距离约束）。没有编码两臂之间科里奥利类耦合的项（`ω₁² sin(θ₁-θ₂)` 等）。没有这些项，驱动混沌的初始条件敏感依赖性就不存在。

3. **均匀的 50/50 分割**：约束求解器在约束的两个粒子之间均匀分配修正。对于链式摆，这意味着内球每次迭代都需要吸收**两个**相邻约束的一半修正——实际上使其"摩擦"加倍。

**解决方案**：完全放弃 Verlet，使用完整的拉格朗日力学 + RK4 积分重写物理引擎。新求解器计算包含三角耦合的完整 N×N 质量矩阵，并通过 Cholesky 分解求解。Verlet → 拉格朗日的过渡是项目历史上**最大的一次重写**。

**教训**：基于约束的物理（Verlet、PBD）引入了数学上不透明的隐式阻尼——你无法轻易量化每帧损失了多少能量。拉格朗日力学虽然在数学上更重，但从设计上就提供了精确的能量守恒特性。

## 2. N 节锁定——刚体缺陷

**严重程度**：致命——导致 N > 2 的摆在物理上不正确。

**症状**：添加第三个关节创建三节摆时，最后两个摆锤"完全锁定在一起，像一个刚体一样运动。"第三个摆锤与第二个共享相同的 θ₂，关节处没有独立弯曲。

**根本原因**：初始 N 节实现将中间粒子**沿杆 2 按比例放置**（与 θ₂ 方向共线），但物理求解器硬编码为 2×2 拉格朗日系统。对于 N > 2 的粒子只有 2 个自由度，额外的关节没有独立的角度自由——它们在几何上与最后一根杆共线。

**修复**：将硬编码的 2×2 求解器替换为**通用的 N 摆拉格朗日求解器**：

```
Mᵢⱼ = lᵢ lⱼ (N−max(i,j)) cos(θᵢ−θⱼ)
bᵢ  = −Σⱼ≠ᵢ Aᵢⱼ sin(θᵢ−θⱼ) ωⱼ² − g lᵢ (N−i) sin(θᵢ)
```

状态现在存储数组 `thetas[]`、`omegas[]`、`ls[]`——每个连接都有自己独立的角度。`computeParticlePositions` 使用简单的前向循环：从固定点开始，将每个摆锤从前一个摆锤的位置以各自 θᵢ 角度放置。

**教训**：当从 N=2 推广到 N>2 时，架构必须从一开始就是 **N 参数化的**。事后添加自由度（例如"只需沿杆 2 插值"）会产生仅凭视觉观察几乎无法诊断的耦合产物。

## 3. 能量守恒——PE 公式符号错误

**严重程度**：高——341% 的能量漂移使能量图失去意义。

**症状**：总能量在 -29 到 +113 焦耳之间振荡，而非保持恒定。

**根本原因**：势能公式使用了 y 轴向下画布惯例（`PE = g Σ (N−i) lᵢ cos(θᵢ)`），但物理引擎的拉格朗日函数使用 y 轴向上惯例。在运动方程中，重力项为 `−g (N−i) lᵢ sin(θᵢ)`，这与欧拉-拉格朗日方程仅在 `V = −g Σ (N−i) lᵢ cos(θᵢ)` 时才一致——注意其中的**取反**。

之前：`PE =  g Σ (N−i) lᵢ cos(θᵢ)` → totalE 漂移 341%
之后：`PE = −g Σ (N−i) lᵢ cos(θᵢ)` → totalE 恒定至 0.0000%

**修复**：一个字符的更改（`g` → `-g`），只需几秒钟输入，但花费了数天调试才发现。

**教训**：坐标系约定在物理模拟中极为重要。画布 y 轴（向下为正）与传统物理 y 轴（向上为正）相反。每个涉及垂直方向的公式都必须审计符号一致性。

## 4. CSS 优先级冲突——代码块渲染

**严重程度**：中——文档显示异常。

**症状**：指南弹窗中的围栏代码块显示为单行，ASCII 目录树中的所有换行符被剥离。

**根本原因**：`#help-modal-content code { white-space: nowrap }`（优先级 1,0,0,1）适用于弹窗中**所有** `<code>` 元素，包括 `<pre>` 块内部的代码。基于类的 `.help-readme-body pre code { white-space: pre }` 规则（优先级 0,2,0,2）被覆盖，因为 **CSS 优先级中 ID 选择器优于类选择器**，不受源代码顺序影响。

在样式表中移动规则**没有任何效果**——这是一个常见的 CSS 误解。

**修复**：在 `pre code` 选择器中添加父级 ID：
```css
#help-modal-content .help-readme-body pre code {
    white-space: pre;
}
```
`#help-modal-content .help-readme-body pre code`（优先级 1,1,2,1）胜过了 `#help-modal-content code`（1,0,0,1）。

**教训**：CSS 优先级是一个**权重系统**，而非顺序系统。`#id` 始终高于 `.class`。覆盖基于 ID 的规则时，修复方法是（a）通过包含父 ID 来增加选择器的优先级，或（b）使用 `!important`（脆弱）。仅重新排序 CSS 规则无济于事。

## 5. GitHub Pages 部署崩溃

**严重程度**：致命——应用在生产环境中完全无法运行。

**症状**：GitHub Pages 上显示空白暗色页面，控制台报错 `TypeError: Cannot read properties of null (reading 'addEventListener')`。动画循环从未启动。

**根本原因**：两个问题叠加：

1. **缺少 `defer`**：`<script>` 标签没有 `defer` 属性，因此浏览器在 HTML 解析过程中同步执行它。在本地（file:// 或 live server），完整 HTML 瞬间到达，掩盖了问题。在 GitHub Pages（CDN 交付、分块传输）上，脚本在某些 DOM 节点**存在之前**就执行了。

2. **ID 不匹配**：某个 `document.getElementById()` 调用返回 `null`，因为元素 ID 在 HTML 中被重命名但在 JS 中未更新（或反之）。对 `null` 调用 `.addEventListener()` 抛出 TypeError，该异常从顶层脚本传播出去，停止了**所有**后续代码——包括整个动画循环。

**修复**：
- 在 `<script>` 标签上添加 `defer`
- 创建 `$(id)` 辅助函数（`document.getElementById` 别名）和 `on(id, event, handler)` 双用途函数，在元素不存在时静默跳过绑定
- 将所有 18 处 `document.getElementById(id).addEventListener(event, handler)` 调用转换为 `on(id, event, handler)`

**教训**：本地开发可能隐藏与时机相关的缺陷。始终在 `file://`、HTTP 和 CDN 服务三种方式下测试。`defer` 属性应是任何引用 DOM 元素的 `<script>` 的**默认选择**。

## 6. 移动端触控弹跳（iOS Safari）

**严重程度**：高——iOS 上完全无法拖拽。

**症状**：在 iPhone 上拖拽摆锤时，页面会弹性滚动而非移动摆锤。

**根本原因**：iOS Safari 的弹性过度滚动和触摸手势处理。即使设置了 `overflow: hidden`，Safari 仍会拦截触摸事件用于页面滚动。原始的 `{ passive: true }` 触摸事件监听器意味着 `preventDefault()` 被浏览器**静默忽略**。

**修复**（三层防御）：

1. CSS：在两个画布上设置 `touch-action: none`——在 GPU 级别告知合成器永远不要将这些元素上的触摸解释为滚动/缩放手势
2. CSS：在 `html, body` 上设置 `overscroll-behavior: none`——在文档级别阻止弹性弹跳
3. JS：将触摸监听器改为 `{ passive: false }`，并在 `dragActive === true` 时有条件地调用 `e.preventDefault()`

**教训**：移动端浏览器触摸处理需要纵深防御。CSS 级别的 `touch-action: none` 最可靠（在合成器线程上运行），但仍需要 JS 级别的 `passive: false` + `preventDefault()` 实现完全控制。这些是互补的，而非冗余的。

## 7. 启动时画布空白

**严重程度**：高——初始加载时摆体不可见。

**症状**：移除 slowMo 模式代码后，启动时摆体不可见——只显示暗色背景。

**根本原因**：两个独立的问题：

1. **缺少同步初始绘制**：引导序列通过 `requestAnimationFrame` 调用 `animate()`，约 16ms 后在下一个 VSync 触发。在此间隙中，画布是透明的（未绘制）。

2. **不安全的 `matchMedia` 使用**：`window.matchMedia("...")` 可能在某些环境中抛出异常，导致顶层脚本在该行停止。

**修复**：
- 添加 `window.matchMedia ? ... : false` 空守卫
- 在引导过程中、`animate()` 之前添加了**同步** `draw()` 和 `updateAngleDisplay()` 调用——摆体在第一个布局帧上即渲染

**教训**：`requestAnimationFrame` 回调不是即时的。如果应用依赖立即出现的视觉效果（例如在指标观测工具捕获页面之前），需要同步绘制路径。此外：始终守卫 `matchMedia`——它并非在所有 JS 环境中都可用。

## 8. 多摆指标——全局数组缺陷

**严重程度**：高——切换摆时相图和能量图显示错误数据。

**症状**：有多个摆时，切换选择或生成新摆会导致图表显示过时/错误的数据。

**根本原因**：指标系统使用单一的全局 `metricsData` 数组，该数组属于收集时恰好为 `selectedPendulum` 的摆。切换选择时，图表渲染器读取的是**前一个**摆的收集数据。

**修复**：每个摆拥有自己的 `metrics[]` 数组，在 `createPendulum()` 中初始化。`collectMetrics()` 遍历所有可见摆：

```js
function collectMetrics() {
    for (const p of pendulums) {
        if (!p.visible) continue;
        p.metrics.push({ thetas, omegas, KE, PE, totalE });
        while (p.metrics.length > METRICS_CAPACITY) p.metrics.shift();
    }
}
```

**教训**：当数据是每个实体独立时，存储也应该是每个实体独立的。全局缓冲区在收集和渲染之间创建了隐藏的耦合，这种耦合仅在特定的交互序列下才会显现。

## 9. 右键点击导致上下文菜单消失

**严重程度**：中——令人困惑的 UI 行为。

**症状**：在画布上任意位置右键点击会使上下文菜单消失。

**根本原因**：`canvasB` 的 `mousedown` 处理程序没有检查 `e.button`。在空白处的右键点击（button 2）触发了 `selectPendulum(null)`，隐藏了上下文菜单。

**修复**：在处理程序顶部添加 `if (e.button !== 0) return;`——只有左键点击触发选择/取消选择。同时添加 `contextmenu` 事件阻止，以抑制浏览器原生的右键菜单。

**教训**：鼠标事件会为所有按钮触发，除非显式过滤。`event.button` 属性中 0 为主键、1 为中键、2 为副键（右键）。当点击有副作用时始终检查它。

## 10. 相图环绕线条产物

**严重程度**：中——相图上出现误导性视觉产物。

**症状**：当 θ₁ 从约 359° 回绕到约 1°（跨越 0°/360° 边界）时，一条水平直线穿过整个相图。

**根本原因**：将所有 θ₁ 值归一化到 [0, 2π) 后，位于范围两端的连续数据点（359° 和 1°）被一条直线连接，因为路径绘制代码对角度环绕一无所知。

**修复**：`drawFadingLine` 现在检测连续 X 值的跳跃是否超过数据范围的一半（`wrapThreshold = range × 0.5`）。检测到此类跳跃时，当前路径被描边并在当前点开始**新**路径——防止跨图线条。

**教训**：角度数据具有线性数据中没有的拓扑特性（模运算）。任何绘制角度量的可视化都必须处理周期性边界条件。

## 11. 角度显示点击被拦截

**严重程度**：中——分析面板打开时可点击摆选择器失效。

**症状**：在分析面板打开或图表放大时，点击角度显示中的摆条目没有反应。

**根本原因**：两个叠加问题：
1. 图表画布的 `click` 处理程序调用了 `e.stopPropagation()`，阻止了 `click` 事件到达角度显示的 `click` 监听器
2. 放大的图表框位于 `z-index: 10`，在物理上覆盖了角度显示区域，拦截了指针事件

**修复**：从 `click` 事件委托改为 `pointerdown`——一种不受 `click` 处理程序的 `stopPropagation()` 影响的不同事件类型：
```js
document.addEventListener('pointerdown', (e) => {
    const entry = e.target.closest('.pend-entry');
    if (!entry || !document.getElementById('angle-display').contains(entry)) return;
    // 选择摆...
});
```

**教训**：`click` 是一个组合事件（同一元素上的 mousedown + mouseup），可能被其任一组件事件的 `stopPropagation()` 阻止。`pointerdown` 在按下时立即触发，不受下游 `click` 处理程序的干扰。对于需要在覆盖类元素下工作的 UI，`pointerdown` 更可靠。

## 12. 能量图 X 轴停滞

**严重程度**：中——图表外观具有误导性。

**症状**：指标环形缓冲区填满（300 点）后，X 轴标签永远不变——卡在 0、60、120、180、240、299——造成时间已停止的假象。

**根本原因**：原始代码使用数组索引作为 X 值（`(_, i) => i`），`xRange.max = data.length - 1`。使用环形缓冲区时，`data.length` 一旦填满就保持不变，因此 X 范围永远不会前进。

**修复**：添加全局 `globalMetricsStep` 计数器，每物理帧递增。每个指标条目存储其步进编号。能量图的 X 范围来自第一个和最后一个条目的步进编号：`{ min: first.step, max: last.step }`。

**教训**：环形缓冲区将数据计数与时间进程解耦。在绘制滚动窗口数据时，对 X 轴使用单调递增的序列号（而非数组索引）。

## 13. 空白图表画布——索引传递缺陷

**严重程度**：中——图表渲染为空。

**症状**：相图和能量图画布完全空白，控制台无错误消息。

**根本原因**：`drawSolidLine` 以单参数调用 `getX(data[i])`，但 X 获取器是 `(_, i) => i`——期望**第二个**参数作为索引。没有它时，`i` 为 `undefined`，坐标映射为每个 X 坐标产生 `NaN`。路径无法绘制任何内容。

```js
// 缺陷：
const px = mapX(getX(data[i]), xMin, xMax, PLOT_W);
// getX = (_, i) => i  →  以 getX(data[i], undefined) 方式调用  →  返回 undefined

// 修复：
const px = mapX(getX(data[i], i), xMin, xMax, PLOT_W);
```

**教训**：JavaScript 函数静默接受任意数量的参数，并会在缺少参数时愉快地传递 `undefined`。当使用接受 `(value, index)` 的工厂风格获取器函数时，始终显式传递两个参数——不要假设索引是隐式的。

## 14. 极端角度下 Cholesky 正定性问题

**严重程度**：低（由安全网处理）——但可能导致静默数据损坏。

**症状**：在极端角度和高 N（7-8 链接）下，质量矩阵理论上可能失去对称正定性，导致对角元上的 `Math.sqrt()` 返回 NaN。

**根本原因**：浮点数范围极限处的数值精度损失。虽然数学上的质量矩阵始终是 SPD（类 Gram 结构），但浮点舍入可能产生微小的负对角元。

**解决方案**：有意**不**在求解器级别处理。NaN 通过物理状态传播，在一帧后被 `isPendulumInvalid()` / `safeResetPendulum()` 捕获——与捕获所有其他数值错误的安全网相同。摆进行"硬重置"（以 135° 重新出现），这在视觉上与现有的自动恢复行为无法区分。

**教训**：并非每个数值边缘情况都需要数学修复。有时系统级安全网是更便宜、更稳健的解决方案——尤其是在触发条件罕见、瞬态且自纠正的情况下。

## 15. 跨变更点的 Float64Array 同步

**严重程度**：低——缺陷仅在特定时机下显现。

**症状**：添加或移除关节后，物理求解器可能读取到过时的 `ls[]` 值。

**根本原因**：`p.ls` 的 `p.ls_f64` Float64Array 镜像需要在 `p.ls` 每次变更时同步。遗漏任何一个同步点都会导致热路径求解器读取错误的杆长——一个静默的正确性缺陷。

**修复**：在所有 4 个变更点添加 `p.ls_f64.set(p.ls)`：
- `addJoint()`
- `removeJoint()`
- `rebuildChain()`（带 `if (p.ls_f64)` 向后兼容守卫）
- `resizeCanvas()`

**教训**：将可变状态镜像到类型数组中会创建一个必须精心维护的同步面。每个修改源数组的代码路径也必须更新镜像。回顾来看，使用辅助函数（`syncLsF64(p)`）会更简洁。

## 16. Verlet 重力太弱

**严重程度**：中——摆体看似冻结。

**症状**：在第七阶段的 Verlet 实现中，摆体几乎不动——"冻结"是用户的准确描述。

**根本原因**：Verlet 重力 `G × pxPerUnit` 每子步长仅产生约 0.022 像素的位移。使用 4 个子步长时，每帧仅约 0.09 像素——视觉上无法察觉。

**修复**：添加 `VERLET_G_SCALE = 8` 倍率放大重力。这是权宜之计——正确的修复是用拉格朗日力学重写物理引擎（第八阶段）。

**教训**：Verlet 积分适用于力与约束力相当的粒子模拟，但对于重力是驱动力的摆系统，像素空间与物理空间之间的比例失配需要精心调参。拉格朗日方法通过使用一致的物理单位完全避免了这个问题。

---

## 17. 关节修改后的陈旧轨迹

**严重程度**：低——视觉闪烁。

**症状**：添加或移除关节后，轨迹渲染系统抛出错误，因为轨迹数组数量与粒子数量不匹配。

**根本原因**：原始的 `addJoint` / `removeJoint` 函数修改了 `particles[]` 和 `constraints[]`，但没有触及 `trails[]` 数组。调整大小后，渲染器期望 `N` 个轨迹数组，但发现了 `N-1`（或 `N+1`）个。

**修复**：`addJoint` 为新粒子压入一个空轨迹数组；`removeJoint` 弹出最后一个轨迹数组。窗口大小变化时也会清除轨迹。

**教训**：当数据模型具有并行数组（粒子、约束、轨迹）时，任何调整大小的操作都必须原子地更新**所有**数组——或者更好地，将它们封装到单个对象中。并行数组是一个反模式，本项目已为此付出了多次代价。

---

## 18. 切换选择时能量图标题未更新

**严重程度**：低——装饰性问题但令人困惑。

**症状**：切换分析面板跟踪的摆时，相图标题仍显示前一个摆的索引，直到数据收集产生新条目。

**根本原因**：标题在 `if (p.metrics.length < 2) return;` 数据守卫**之后**更新。如果新选中的摆还没有指标数据，标题永远不会更新。

**修复**：将标题更新移至 `renderPhasePortrait()` 的开头，在数据可用性检查之前。

**教训**：从函数提前返回的守卫不应阻止揭示状态的更新（如标题）。安排操作顺序，使元数据更新在数据相关操作之前发生。
