## 1. Verlet 重力太弱 —— 摆冻住了

**阶段**：第七阶段（Verlet 原型）
**严重程度**：中
**症状**：摆看起来冻住了。

**分析**：Verlet 重力的计算公式为 `y += vy + G * pxPerUnit * h²`。代入 `G = 9.81`、`pxPerUnit ≈ 60`、`h = 1/240 s`，每个子步长的位移约为 **0.022 像素**。每帧 4 个子步长，合计约 0.09 像素/帧——肉眼不可见。根本原因是单位比例失配：Verlet 将重力直接作用于像素坐标，而基于角度的 RK4 产生角加速度，经杆长放大后形成可观的线加速度。

**解决**：引入缩放因子 `VERLET_G_SCALE = 8` 作为临时措施。最终方案——以完整拉格朗日力学替代 Verlet——在第八阶段实现。

**教训**：在 Verlet 积分中，必须先验证 `G * pxPerUnit * h²` 的量级，再排查系统看似无响应的其他原因。

## 2. 关节修改后轨迹不同步

**阶段**：第七阶段（Verlet 原型）
**严重程度**：低
**症状**：视觉闪烁；添加关节时控制台报错。

**分析**：`addJoint` 向 `particles[]` 和 `constraints[]` 追加了新条目，但遗漏了 `trails[]`。`removeJoint` 则出现了相反的遗漏。并行数组设计（`particles[]`、`constraints[]`、`trails[]`）要求三个数组原子同步更新，但这一点未得到强制执行。

**解决**：在 `addJoint` 中添加 `p.trails.push([])`，在 `removeJoint` 中添加 `p.trails.pop()`。`resizeCanvas` 中也增加了全量轨迹清除逻辑。

**教训**：并行数组是一种反模式。将相关数据封装为单一对象可消除整类同步缺陷。

## 3. Verlet 数值阻尼 —— 混沌消失

**阶段**：第七阶段→第八阶段（物理引擎重写）
**严重程度**：致命
**症状**：摆呈现规则、可预测的运动；未观察到混沌分叉。

**分析**：三个因素叠加：
1. **约束迭代开销**：4 子步长 × 10 次迭代 = 每帧 40 次位置修正，每次将动能弯曲为位置修正，持续耗散能量。
2. **缺失非线性耦合**：Verlet 将每个粒子独立处理；缺乏驱动初始条件敏感依赖的科里奥利类耦合项（`ω₁² sin(θ₁−θ₂)`）。
3. **内球双重摩擦**：50/50 约束修正分割使连接两个约束的关节有效阻尼加倍。

能量监测确认总能量在 5 秒内从约 30 J 降至 10 J 以下。

**解决**：使用完整拉格朗日力学、RK4 积分、Cholesky 分解和 N×N 质量矩阵重写了整个物理引擎。能量漂移从数十个百分点降至 **0.0000%**。

**教训**：基于约束的物理引入了数学上不透明的隐式阻尼。对于能量守恒至关重要的混沌系统，拉格朗日力学不仅是更优选择——它是必需的。

## 4. N 节锁定 —— 刚体缺陷

**阶段**：第八阶段（拉格朗日重写）
**严重程度**：致命
**症状**：当 N > 2 时，摆锤完全平行运动；最后两个摆锤表现为单一刚体。

**分析**：两个缺陷共享同一根本原因：
1. `computeParticlePositions` 将摆锤 3 置于角度 θ₂（继承摆锤 2 的角度），而非赋予其独立的 θ₃。
2. 求解器硬编码为 2×2 系统，没有 θ₃、θ₄ 等的方程。

**解决**：以通用 N 摆拉格朗日求解器替代硬编码的 2×2 求解器，构建 N×N 质量矩阵。每个连接现在在 `thetas[]` 中维护独立的角度。

**教训**：从 N=2 推广到 N>2 要求架构从一开始就参数化为 N。事后补加自由度会产生几乎无法仅凭肉眼诊断的耦合伪影。

## 5. 右键点击取消上下文菜单

**阶段**：第八阶段（交互优化）
**严重程度**：中
**症状**：在画布上右键点击导致上下文菜单消失。

**分析**：`canvasB` 的 mousedown 处理程序未过滤 `e.button`。JavaScript 的点击事件对所有鼠标按钮都会触发；`selectPendulum(null)` 在任何点击时都被调用，包括右键点击（`e.button === 2`）。

**解决**：在处理程序入口添加 `if (e.button !== 0) return;`，并在 `contextmenu` 事件中调用 `e.preventDefault()` 以抑制浏览器原生菜单。

**教训**：画布事件处理程序应始终对非左键点击进行过滤。`e.button !== 0` 应作为所有 mousedown 处理程序的标准模板。

## 6. 能量不守恒 —— 势能公式符号错误

**阶段**：第八阶段后期（指标实现）
**严重程度**：高
**症状**：341% 能量漂移；总能量在 −29 J 到 +113 J 之间振荡。

**分析**：势能公式为 `PE = g * Σᵢ (N−i) * lᵢ * cos(θᵢ)`，这在画布坐标系（y 轴向下）中看似直观，但与物理惯例（y 轴向上）不一致。拉格朗日方程要求 `V = −g * Σᵢ (N−i) * lᵢ * cos(θᵢ)`。缺失的负号导致动能与势能同相振荡，而非反相。

**解决**：将 `PE = g * sum` 改为 `PE = -g * sum`。

**教训**：坐标系约定在物理模拟中具有根本性意义。每个涉及垂直方向的公式都必须审计画布坐标与物理坐标之间的符号一致性。

## 7. 空白图表 —— Getter 参数缺失

**阶段**：第八阶段后期（指标渲染）
**严重程度**：中
**症状**：能量图和相图均渲染为空白黑色画布，控制台无报错。

**分析**：`drawSolidLine` 以单参数调用 `getX(data[i])`，但能量图的 getter 签名为 `(_, i) => i`，需要索引作为第二参数。缺失的参数产生 `undefined`，经 `mapX` 转换为 `NaN`。Canvas 的 `lineTo` 对 `NaN` 坐标静默忽略。相图的 getter（`d => d.thetas[0]`）仅使用第一参数，未触发该缺陷，导致调试方向被误导至数据收集而非渲染。

**解决**：将 `getX(data[i])` 改为 `getX(data[i], i)`。

**教训**：JavaScript 灵活的参数处理虽便利，但在工厂函数传递参数时会产生静默错误。签名为 `(value, index)` 的 API 必须始终接收两个参数。

## 8. 多摆指标 —— 全局数组耦合

**阶段**：第八阶段后期（多摆支持）
**严重程度**：高
**症状**：切换摆时图表显示错误数据。

**分析**：`collectMetrics()` 根据当前选中的摆向单一全局 `metricsData` 数组填充数据。切换选择时，多个摆的数据交错存入同一数组，且无机制区分来源。新生成的摆因尚未被选中而没有数据。

**解决**：每个摆现在拥有独立的 `metrics[]` 数组。`collectMetrics()` 遍历所有可见的摆，图表渲染器从选中摆的缓冲区读取数据。

**教训**：当数据是每实体独立时，存储也必须是每实体独立的。全局缓冲区在数据收集与渲染之间创建隐藏耦合，这种耦合只在特定交互序列下才会暴露。

## 9. 能量图 X 轴停滞

**阶段**：第八阶段后期（指标细化）
**严重程度**：中
**症状**：300 点环形缓冲区存满后，X 轴刻度冻结在 299。

**分析**：X getter 使用数组索引 `(_, i) => i`，且 `xRange.max = data.length - 1`。在环形缓冲区中，一旦存满，`data.length` 固定为容量上限，使数组长度与时间进程解耦。

**解决**：引入单调递增的 `globalMetricsStep` 计数器。每个指标条目保存其步数；能量图使用 `d => d.step` 作为 X getter，范围由 `first.step` 和 `last.step` 推导。

**教训**：环形缓冲区将数据数量与时间进程解耦。`array.length` 不是时间的代理。滚动窗口可视化需要外部单调序列号作为 X 轴。

## 10. 相图环绕伪影

**阶段**：第八阶段后期（指标细化）
**严重程度**：中
**症状**：摆锤旋转多圈后，相图上出现一条贯穿全图的水平直线。

**分析**：θ₁ 被归一化到 [0°, 360°)。跨越 0°/360° 边界时（如 359° → 1°），相邻点在角度空间中仅差 2°，但在绘图空间中相隔 358°。路径绘制器用直线连接它们，形成横跨全图的线段。

**解决**：在 `drawFadingLine` 中，以数据范围的一半（[0, 2π) 时为 π）作为环绕阈值，当 `|x₂ − x₁| > threshold` 时断开路径。

**教训**：角度数据存在于圆上，而非区间上。任何绘制角度量的可视化都必须显式处理周期性边界条件。模运算下，数据相邻不等于视觉相邻。

## 11. 角度显示点击被拦截

**阶段**：第八阶段后期（指标细化）
**严重程度**：中
**症状**：图表画布打开或放大时，可点击摆选择器无响应。

**分析**：图表画布的 `click` 处理程序调用了 `e.stopPropagation()`，阻止事件冒泡到角度显示的 `click` 监听器。放大后的图表框以 `z-index: 10` 物理覆盖角度显示区域，指针事件被拦截并抑制。

**解决**：以文档级的 `pointerdown` 替代 `click` 事件委托，后者立即触发且不受下游 `stopPropagation()` 影响。

**教训**：`click` 是在 `mousedown` 或 `mouseup` 阶段都可能被 `stopPropagation()` 阻止的合成事件。`pointerdown` 立即触发，对于必须在覆盖层下保持可交互的 UI 更为可靠。

## 12. 切换选择时能量图标题不更新

**阶段**：第八阶段后期（指标细化）
**严重程度**：低
**症状**：切换选择后，相图标题仍显示错误的摆编号。

**分析**：标题更新位于提前返回守卫（`if (p.metrics.length < 2) return;`）之后。切换到刚创建、指标数据不足的新摆时，守卫在标题更新前触发。

**解决**：将标题更新移至数据守卫之前。

**教训**：提前返回守卫不应阻止元数据更新。状态揭示操作（标题、标签、轴范围）应置于数据依赖检查之前。

## 13. CSS 优先级冲突 —— 代码块渲染错乱

**阶段**：后期（Wiki 阅读器）
**严重程度**：中
**症状**：指南弹窗中的围栏代码块全部显示为单行。

**分析**：`#help-modal-content code`（优先级 1,0,0,1）对所有 `<code>` 元素应用了 `white-space: nowrap`，包括 `<pre>` 块内部的代码。试图覆盖的规则 `.help-readme-body pre code`（优先级 0,2,0,2）被击败，因为 ID 选择器无论源码顺序如何始终高于类选择器。

**解决**：将选择器优先级提升至 `#help-modal-content .help-readme-body pre code`（1,1,2,1）。

**教训**：CSS 优先级是权重系统，而非顺序系统。当选择器权重不同时，重排规则毫无意义。覆盖必须匹配或超过目标选择器的优先级。

## 14. 移动端触控弹跳（iOS Safari）

**阶段**：第九阶段（移动端支持）
**严重程度**：高
**症状**：iOS 上无法拖拽摆锤；页面发生弹性滚动。

**分析**：iOS Safari 的弹性过度滚动劫持所有触摸事件。原始触摸监听器使用 `{ passive: true }`，导致浏览器静默忽略 `preventDefault()` 调用。

**解决**：实施三层防御：
1. **合成器级别**：画布上设置 `touch-action: none`。
2. **文档级别**：根元素设置 `overscroll-behavior: none`。
3. **JavaScript 级别**：使用 `{ passive: false }` 监听器，在拖拽激活时有条件地调用 `preventDefault()`。

**教训**：移动端触摸处理需要纵深防御。CSS 的 `touch-action: none`（合成器线程）是最可靠的层级，但 JS 级别的 `passive: false` + `preventDefault()` 仍为实现完全控制所必需。各层级互补，而非冗余。

## 15. 启动时画布空白

**阶段**：第九阶段（移动端支持）
**严重程度**：高
**症状**：初始页面加载时摆不可见。

**分析**：两个独立问题：
1. **缺少同步初始绘制**：`animate()` 通过 `requestAnimationFrame` 排队，将首帧推迟到下一个 VSync（约 16 ms）。此间隙内画布保持透明。移除 slowMo 代码消除了一个偶然的"额外"绘制，暴露了此间隙。
2. **不安全的 `matchMedia`**：`window.matchMedia` 并非在所有环境中都可用（无头浏览器、部分 WebView）。未加保护的调用可能抛出异常并终止整个顶层脚本。

**解决**：添加 `window.matchMedia` 空值保护，并在 `animate()` 之前添加同步的 `draw()` 和 `updateAngleDisplay()` 调用。

**教训**：`requestAnimationFrame` 并非即时执行。需要立即视觉反馈的应用需要同步绘制路径。浏览器 API 的可用性不应被假设。

## 16. GitHub Pages 部署崩溃

**阶段**：第九阶段（部署）
**严重程度**：致命
**症状**：GitHub Pages 上显示空白暗色页面；`TypeError: Cannot read properties of null`。

**分析**：两个问题叠加：
1. **缺少 `defer`**：`<script>` 标签未加 `defer`，导致脚本在所有 DOM 节点解析前执行——本地环境中因文件加载毫秒级完成而从未出现此情况。
2. **ID 不匹配**：`document.getElementById(id).addEventListener()` 在元素为 `null` 时抛出异常（由时机问题或 ID 重命名导致），终止所有后续脚本执行。

**解决**：为 `<script>` 标签添加 `defer`，并创建 `on(id, event, handler)` 安全绑定函数，对缺失元素静默跳过。

**教训**：本地开发环境隐藏了所有与时机相关的缺陷。`defer` 应作为任何引用 DOM 元素的脚本的默认选择。安全绑定模式（`on(id, event, handler)`）可防止对 `null` 调用方法——这是 JavaScript 中最常见且最易被忽视的生产环境崩溃原因之一。

## 17. 极端角度下 Cholesky 分解的正定性问题

**阶段**：后期优化
**严重程度**：低（被安全网捕获）
**症状**：极端角度、高 N 时存在静默数据损坏的风险。

**分析**：数学上 M 始终是对称正定的。然而，在近奇异构型下，浮点舍入可能产生微小的负对角线元素（如 −1e−16），导致 `choleskyFactor` 中 `Math.sqrt()` 返回 NaN。

**解决**：未在求解器层面添加回退逻辑，而是让 NaN 传播一帧后被 `isPendulumInvalid()` / `safeResetPendulum()` 捕获，将摆重置至安全状态（135°、零速度、清除轨迹）。硬上限（`MAX_LINKS = 8`、`MAX_PENDULUMS = 8`）防止进入近奇异区域。

**教训**：并非每个数值边缘情况都需要数学层面的修复。系统级安全网通常是更便宜、更健壮的解决方案——前提是安全网可靠且触发条件罕见、可自我纠正。

## 18. Float64Array 跨变更点同步问题

**阶段**：后期优化
**严重程度**：低（潜在缺陷）
**症状**：物理求解器可能读到过时的杆长。

**分析**：`p.ls_f64` 镜像 `p.ls` 以实现热路径无 GC 访问。四个变更点修改 `p.ls`：`addJoint`、`removeJoint`、`rebuildChain`、`resizeCanvas`。初始实现遗漏了 `resizeCanvas`，该操作改变 `pxPerUnit` 从而改变所有杆长。

**解决**：在所有四个变更点添加 `p.ls_f64.set(p.ls)`，并在 `rebuildChain` 中设置向后兼容保护。

**教训**：将可变状态镜像到类型数组会创建一个需要精心维护的同步面。集中式辅助函数（`syncLsF64`）可将同步逻辑本地化，降低未来遗漏的风险。根本问题是数据冗余——同一份信息存在于两种表示中，需要显式的一致性维护。
