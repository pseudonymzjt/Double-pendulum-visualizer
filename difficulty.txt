# Difficulties I Encountered — Chaotic Art: Double Pendulum Visualizer

> Every bug I hit, every dead end I walked into, and how I eventually got out. There are 18 of them here, roughly ordered by how much they hurt.

---

## 1. Verlet Numerical Damping (Stage 7 → Stage 8 Rewrite)

**How bad**: Critical. The pendulum wouldn't chaos.

**What happened**: My Stage 7 Verlet + constraint-solver approach made the trajectory look "too regular and lacked chaotic movement, even at initial high angle." The user asked if I had small-angle approximations or excessive damping in the code. I had neither — or so I thought.

**Why**: Three problems, all compounding:

1. **40 constraint corrections per frame**: 4 sub-steps × 10 iterations = 40 position adjustments per frame, each splitting corrections 50/50. This acts as implicit damping — every correction bleeds a tiny bit of kinetic energy.

2. **No nonlinear coupling**: Verlet treats each particle independently (gravity + distance constraints). There are no terms encoding the Coriolis-like coupling between arms (`ω₁² sin(θ₁-θ₂)`, etc.). Without those terms, there's no sensitive dependence on initial conditions — the engine of chaos.

3. **Uniform 50/50 split**: The constraint solver distributes corrections evenly. For a chain pendulum, the inner bob absorbs half the correction from **both** adjacent constraints — effectively doubling the friction on that joint.

**How I fixed it**: I abandoned Verlet entirely and rewrote the physics engine using exact Lagrangian mechanics with RK4 integration. The new solver computes the full N×N mass matrix with trigonometric coupling and solves it via Cholesky decomposition. This was the **single largest rewrite** in the project's history.

**What I learned**: Constraint-based physics (Verlet, PBD) introduces implicit damping that's mathematically opaque — you can't easily quantify how much energy you're losing per frame. Lagrangian mechanics is heavier mathematically, but it gives you exact energy conservation by design.

## 2. N-Link Locking — The Rigid-Body Bug

**How bad**: Critical. N > 2 pendulums were physically wrong.

**What happened**: When I added a third joint to make a triple pendulum, the last two bobs were "completely locked together and act like a single rigid body." The 3rd bob shared the same θ₂ — no independent bending at the joint.

**Why**: My initial N-link approach placed intermediate particles **proportionally along rod 2** (collinear with the θ₂ direction), while the physics solver was hardcoded as a 2×2 Lagrangian system. With only 2 degrees of freedom for N > 2 particles, the extra joints had no angular freedom — they were geometrically collinear with the last rod.

**How I fixed it**: I replaced the hardcoded 2×2 solver with a **general N-pendulum Lagrangian solver**:

```
Mᵢⱼ = lᵢ lⱼ (N−max(i,j)) cos(θᵢ−θⱼ)
bᵢ  = −Σⱼ≠ᵢ Aᵢⱼ sin(θᵢ−θⱼ) ωⱼ² − g lᵢ (N−i) sin(θᵢ)
```

Each link now has its own independent angle in `thetas[]`, and `computeParticlePositions` just walks from the pivot placing each bob at its own θᵢ.

**What I learned**: When you generalize from N=2 to N>2, the architecture must be **N-parameter from the start**. Adding degrees of freedom as an afterthought ("just interpolate along rod 2") creates coupling artifacts that are nearly impossible to diagnose visually.

## 3. Energy Conservation — Wrong Sign in PE Formula

**How bad**: High. 341% energy drift made the energy plot useless.

**What happened**: Total energy oscillated between -29 and +113 joules instead of staying constant.

**Why**: I used the canvas y-down convention (`PE = g Σ (N−i) lᵢ cos(θᵢ)`) but the Lagrangian equations use y-up. In the equations of motion, the gravity term is `−g (N−i) lᵢ sin(θᵢ)`, which matches Euler-Lagrange only when `V = −g Σ (N−i) lᵢ cos(θᵢ)` — note the **negation**.

Before: `PE =  g Σ (N−i) lᵢ cos(θᵢ)` → totalE drifts 341%
After:  `PE = −g Σ (N−i) lᵢ cos(θᵢ)` → totalE constant to 0.0000%

**How I fixed it**: One character. `g` → `-g`. Took seconds to type, days to discover.

**What I learned**: Coordinate system conventions matter profoundly in physics simulation. The canvas y-axis (down = positive) is opposite to physics (up = positive). Every formula that references vertical direction needs sign auditing.

## 4. CSS Specificity War — Code Block Rendering

**How bad**: Medium. Documentation display was broken.

**What happened**: Fenced code blocks in the Guide modal rendered as a single line. The ASCII directory tree was flat.

**Why**: `#help-modal-content code { white-space: nowrap }` (specificity 1,0,0,1) applied to ALL `<code>` elements, even those inside `<pre>` blocks. My class-based `.help-readme-body pre code { white-space: pre }` (specificity 0,2,0,2) was overridden because **ID selectors beat class selectors**, regardless of source order. I tried moving rules around in the stylesheet — **zero effect**.

**How I fixed it**: I added the parent ID to the `pre code` selector:
```css
#help-modal-content .help-readme-body pre code {
    white-space: pre;
}
```
`#help-modal-content .help-readme-body pre code` (specificity 1,1,2,1) beats `#help-modal-content code` (1,0,0,1).

**What I learned**: CSS specificity is a **weight system**, not a proximity system. `#id` always beats `.class`. To override an ID-based rule, you either include the parent ID in your selector or use `!important` (but don't). Reordering CSS alone is a waste of time.

## 5. GitHub Pages Deployment Crash

**How bad**: Critical. The app was completely broken in production.

**What happened**: A blank dark page on GitHub Pages. Console: `TypeError: Cannot read properties of null (reading 'addEventListener')`. The animation loop never started.

**Why**: Two issues together:

1. **No `defer`**: The `<script>` tag had no `defer`, so the browser executed it synchronously during HTML parsing. On localhost the full HTML arrived instantly — never a problem. On GitHub Pages (CDN-served, chunked), the script ran **before** some DOM nodes existed.

2. **ID mismatch**: A `document.getElementById()` returned `null`. Calling `.addEventListener()` on `null` threw a TypeError that killed the **entire** top-level script — animation loop, controls, everything.

**How I fixed it**:
- Added `defer` to the `<script>` tag
- Created `$(id)` helper and `on(id, event, handler)` — silently skips binding if the element doesn't exist
- Converted all 18 raw `document.getElementById(id).addEventListener(...)` calls to `on(id, event, handler)`

**What I learned**: Local development hides timing bugs. Always test `file://` vs HTTP vs CDN. And `defer` should be the **default** for any `<script>` that touches the DOM.

## 6. Mobile Touch Bounce (iOS Safari)

**How bad**: High. Dragging was impossible on iOS.

**What happened**: On iPhone, trying to drag a pendulum bob would scroll the page instead.

**Why**: iOS Safari's elastic overscroll hijacks touch events. Even with `overflow: hidden`, Safari intercepts them for scrolling. My original `{ passive: true }` listeners meant `preventDefault()` was **silently ignored**.

**How I fixed it** (three layers):

1. CSS: `touch-action: none` on both canvases — tells the compositor at the GPU level to never interpret touches as scroll/zoom
2. CSS: `overscroll-behavior: none` on `html, body` — document-level bounce prevention
3. JS: Changed to `{ passive: false }`, called `e.preventDefault()` conditionally when actually dragging

**What I learned**: Mobile touch handling needs defense-in-depth. CSS-level `touch-action: none` is the most reliable, but you still need `preventDefault()` with `passive: false` for full control. They're complementary, not redundant.

## 7. Blank Canvas at Startup

**How bad**: High. The pendulum was invisible on initial load.

**What happened**: After I removed some slowMo-related code, the pendulum stopped appearing at startup. Just a dark background.

**Why**: Two independent problems:

1. **No synchronous initial paint**: My bootstrap called `animate()` via `requestAnimationFrame`, which fires ~16ms later at the next VSync. During that gap, the canvases were transparent.

2. **Unsafe `matchMedia`**: `window.matchMedia("...")` could throw in some environments, killing the script at that line.

**How I fixed it**:
- Added `window.matchMedia ? ... : false` null guard
- Added **synchronous** `draw()` and `updateAngleDisplay()` calls before `animate()` — the pendulum renders on the very first layout frame

**What I learned**: `requestAnimationFrame` is not instant. If you need an immediate visual, you need a synchronous paint path. And always guard `matchMedia`.

## 8. Multiple Pendulum Metrics — Global Array Bug

**How bad**: High. The phase portrait and energy plots showed wrong data when I switched pendulums.

**What happened**: With multiple pendulums on screen, selecting a different one or spawning a new one caused stale or wrong data to appear in the plots.

**Why**: I had a single global `metricsData` array that belonged to whichever pendulum happened to be selected at collection time. When selection switched, the plot renderers read data from the **previous** pendulum's collection.

**How I fixed it**: Each pendulum now owns its own `metrics[]` array, initialized in `createPendulum()`. `collectMetrics()` iterates all visible pendulums independently.

```js
function collectMetrics() {
    for (const p of pendulums) {
        if (!p.visible) continue;
        p.metrics.push({ thetas, omegas, KE, PE, totalE });
        while (p.metrics.length > METRICS_CAPACITY) p.metrics.shift();
    }
}
```

**What I learned**: When data is per-entity, storage should be per-entity too. Global buffers create hidden coupling that only manifests under specific interaction sequences.

## 9. Right-Click Killing the Context Menu

**How bad**: Medium. Confusing UI behavior.

**What happened**: Right-clicking anywhere on the canvas made the contextual menu disappear.

**Why**: My `canvasB` mousedown handler had no `e.button` check. A right-click (button 2) on empty space triggered `selectPendulum(null)`, which hid the menu.

**How I fixed it**: Added `if (e.button !== 0) return;` at the top of the handler — only left-click triggers selection. Also suppressed the browser's native right-click menu.

**What I learned**: Mouse events fire for all buttons unless you filter them. `event.button` is 0 for left, 1 for middle, 2 for right. Always check it when clicks have side effects.

## 10. Phase Portrait Wrap-Around Line Artifact

**How bad**: Medium. Misleading visual artifact.

**What happened**: When θ₁ wrapped from ~359° back to ~1° (crossing the 0°/360° boundary), a straight line appeared across the entire phase portrait.

**Why**: I normalized all θ₁ values to [0, 2π), but consecutive data points at opposite ends of the range were connected by a direct line — the path drawer didn't know about angular wrapping.

**How I fixed it**: `drawFadingLine` now detects consecutive X values that jump by more than half the range (`wrapThreshold = range × 0.5`). When detected, it strokes the current path and starts a **new** one at the current point.

**What I learned**: Angular data has topological quirks that linear data doesn't. Any plot of angular quantities has to handle the periodic boundary condition.

## 11. Angle Display Click Interception

**How bad**: Medium. The clickable pendulum selector stopped working when plots were open.

**What happened**: Clicking a pendulum entry in the angle display did nothing when the metrics panel was open or a plot was zoomed.

**Why**: Two problems:
1. Plot canvas `click` handlers called `e.stopPropagation()`, blocking `click` from reaching my listener
2. Zoomed plot boxes at `z-index: 10` physically covered the angle display area

**How I fixed it**: Switched from `click` event delegation to `pointerdown` — a different event type not affected by `click` handlers' `stopPropagation()`.

```js
document.addEventListener('pointerdown', (e) => {
    const entry = e.target.closest('.pend-entry');
    if (!entry || !document.getElementById('angle-display').contains(entry)) return;
    // select pendulum...
});
```

**What I learned**: `click` is a composed event (mousedown + mouseup) and can be blocked by `stopPropagation()` on either component. `pointerdown` fires immediately and can't be blocked by downstream `click` handlers. For overlay-resistant UI, `pointerdown` is more reliable.

## 12. Energy Plot X-Axis Stagnation

**How bad**: Medium. The plot looked frozen.

**What happened**: Once the metrics ring buffer filled (300 points), the X-axis labels never changed — stuck at 0, 60, 120, 180, 240, 299. It looked like time had stopped.

**Why**: I used array index as X value (`(_, i) => i`) with `xRange.max = data.length - 1`. With a ring buffer, `data.length` stays constant once full, so the X range never advances.

**How I fixed it**: Added a global `globalMetricsStep` counter that increments each physics frame. Each entry stores its step number. The energy plot's X range comes from the first and last entries' step numbers.

**What I learned**: Ring buffers decouple data count from time progression. For rolling window plots, use monotonic sequence numbers, not array indices.

## 13. Blank Plot Canvas — Index Pass-Through Bug

**How bad**: Medium. Both plots rendered nothing.

**What happened**: Phase portrait and energy plot canvases were completely blank. No console errors.

**Why**: `drawSolidLine` called `getX(data[i])` with a single argument, but the X getter was `(_, i) => i` — expecting a **second** argument for the index. Without it, `i` was `undefined`, and every X coordinate became `NaN`.

```js
// Bug:
const px = mapX(getX(data[i]), xMin, xMax, PLOT_W);
// getX = (_, i) => i  →  called as getX(data[i], undefined)  →  undefined

// Fix:
const px = mapX(getX(data[i], i), xMin, xMax, PLOT_W);
```

**What I learned**: JavaScript functions accept any number of arguments and happily pass `undefined` for missing ones. When using factory-style getters that accept `(value, index)`, always pass both explicitly.

## 14. Cholesky Positive-Definiteness at Extreme Angles

**How bad**: Low (caught by the safety net). Could have caused silent corruption.

**What happened**: At extreme angles and high N (7-8 links), the mass matrix could theoretically lose symmetric positive-definiteness, causing `Math.sqrt()` on the diagonal to return NaN.

**Why**: Floating-point rounding at the edge of the representable range. The mathematical matrix is always SPD, but a tiny negative diagonal element can appear due to precision loss.

**How I handled it**: I intentionally **didn't** fix this at the solver level. The NaN propagates into the physics state and gets caught one frame later by `isPendulumInvalid()` / `safeResetPendulum()` — the same safety net that catches everything else. The pendulum reappears at 135° on the next frame.

**What I learned**: Not every numerical edge case needs a mathematical fix. Sometimes the system-level safety net is cheaper and more robust — especially when the triggering conditions are rare, transient, and self-correcting.

## 15. Float64Array Sync Across Mutation Points

**How bad**: Low. Would only manifest under specific timing.

**What happened**: After adding or removing a joint, the physics solver could read stale `ls[]` values.

**Why**: My `p.ls_f64` Float64Array mirror of `p.ls` needed updating every time `p.ls` changed. Miss a sync point, and the hot-path solver silently reads wrong link lengths.

**How I fixed it**: Added `p.ls_f64.set(p.ls)` at all 4 mutation points:
- `addJoint()`
- `removeJoint()`
- `rebuildChain()` (with `if (p.ls_f64)` guard for backward compat)
- `resizeCanvas()`

**What I learned**: Mirroring mutable state into a typed array creates a synchronization surface you have to maintain meticulously. Every path that modifies the source must also update the mirror. In hindsight, a helper function like `syncLsF64(p)` would have been cleaner.

## 16. Verlet Gravity Too Weak

**How bad**: Medium. The pendulum appeared frozen.

**What happened**: In the Stage 7 Verlet implementation, the pendulum barely moved. "Frozen" was the exact word.

**Why**: Verlet gravity `G × pxPerUnit` gave ~0.022 pixels of displacement per sub-step. With 4 sub-steps, that's ~0.09 px/frame — visually imperceptible.

**How I fixed it**: Added `VERLET_G_SCALE = 8` as a multiplier. It was a band-aid — the real fix was rewriting with Lagrangian mechanics (Stage 8).

**What I learned**: Verlet works well when forces and constraints are comparable, but for pendulums where gravity is the driver, the pixel-space/physics-space scale mismatch requires careful tuning. Lagrangian mechanics avoids this by working in consistent physical units.

## 17. Stale Trails on Joint Modification

**How bad**: Low. Visual glitch.

**What happened**: After adding or removing a joint, trail rendering threw errors because the trail array count didn't match the particle count.

**Why**: `addJoint` / `removeJoint` modified `particles[]` and `constraints[]` but not `trails[]`. After resizing, the renderer expected N trail arrays but found N-1 (or N+1).

**How I fixed it**: `addJoint` pushes an empty trail array; `removeJoint` pops the last one. Trails also get cleared on window resize.

**What I learned**: Parallel arrays (particles, constraints, trails) are an anti-pattern. Any resize operation has to update **all** of them atomically — or better, encapsulate them into a single object. This project has paid for this design choice multiple times.

---

## 18. Energy Plot Title Not Updating on Selection Switch

**How bad**: Low. Cosmetic, but confusing.

**What happened**: When switching which pendulum the metrics panel tracked, the phase portrait title still showed the previous pendulum's index until new data came in.

**Why**: I updated the title **after** the `if (p.metrics.length < 2) return;` guard. If the new pendulum had no data yet, the title never updated.

**How I fixed it**: Moved the title update to the top of `renderPhasePortrait()`, before any data availability check.

**What I learned**: Guards that early-return shouldn't prevent metadata (like titles) from updating. Order operations so that state-revealing updates happen before data-dependent ones.
