## 1. 概念层面的创新

### 1.1 物理精度：不妥协

现有的双摆可视化器大致可分为两类：
- **欧拉积分** + 小角度近似——计算速度快，但精度不足，无法呈现真正的混沌行为。
- **Verlet 约束求解**——稳定性良好，但求解器本身引入了隐式阻尼，持续耗散能量，最终使系统退化为周期性运动。

本项目采用了根本不同的技术路线：直接运用**完整的拉格朗日力学**，构建 N×N 质量矩阵并保留所有三角耦合项。不采用 `sin θ ≈ θ` 之类的近似。RK4 积分器每帧执行八个子步长（`h = 1/480 s`），实测数百步后能量漂移为 **0.0000%**。

双摆的混沌动力学源于 `sin(θ₁−θ₂)` 耦合项以及科里奥利类的 `ω₁²·sin(Δ)`、`ω₂²·sin(Δ)` 项。大多数简化求解器舍弃了这些项；本实现完整保留了它们，使蝴蝶效应得以在屏幕上直观呈现。

### 1.2 混沌模式：从"模拟"到"观测"

传统展示混沌的方法是将两个模拟并排运行。本实现允许用户**随时激活混沌模式**，实时观察轨迹分叉。

- 摆 B 在摆 A 的**当前**角度配置上加 0.01° 生成（而非基于某个先前的初始条件）。
- 摆 B **继承摆 A 的完整轨迹历史**，确保两条轨迹共享同一原点。
- 分叉点在视觉上一目了然：青色与品红色轨迹从用户按下 `C` 键的精确时刻开始分离。

这一设计将混沌从纯粹的模拟现象转化为**可观测的体验**。用户可以让系统自由演化，在任意感兴趣的时刻激活混沌，并观察蝴蝶效应从该瞬间展开。

### 1.3 渲染理念：艺术，而非图表

视觉设计优先考虑美学品质，使其足以导出为高分辨率 PNG 作品，而非遵循枯燥的物理图表惯例。

- **批次渐隐轨迹**：80 个不透明度批次（α 从 0.02 到 0.90），产生平滑、有机的渐隐效果，类似霓虹灯光的衰减。
- **速度自适应线宽**：末端轨迹线宽从 0.8 px（高速）到 3.0 px（低速）变化，赋予轨迹画笔笔触般的质感。
- **双层画布架构**：轨迹（A 层）与摆体（B 层）独立渲染，消除视觉残影的同时保持几何清晰度。
- **霓虹辉光**：相图与能量图的线条绘制两次——先以宽而低透明度的描边，再在其上方以窄而明亮的描边——产生发光效果。
- **极坐标 HUD**：拖拽操作时出现量角器风格的叠加层，平滑淡入淡出，兼具功能性与视觉美感。

导出的 PNG 输出更像艺术作品，而非科学图表。

## 2. 技术层面的创新

### 2.1 零分配物理引擎

热路径通过两项互补优化彻底消除了垃圾回收压力。

#### Cholesky 分解（替代高斯消元）

N×N 质量矩阵 M 是对称正定的——这是拉格朗日量天然赋予的类 Gram 结构。Cholesky 分解 `M = L·Lᵀ` 充分利用了这一性质：

| 指标 | 高斯消元（基线） | Cholesky 分解（优化后） |
|------|-----------------|-----------------------|
| 运算量 | ~2N³/3 次浮点运算 | ~N³/3 次浮点运算 |
| 主元选择 | 需要部分主元 | 不需要（SPD 保证） |
| 内存开销 | 行交换开销 | 原地写入暂存缓冲区 |

当 N = 8 时，可获得约 **2 倍加速**，且零额外内存分配。

#### JIT 友好的 Float64Array 暂存缓冲区

原始 `rk4Step()` 实现每子步长通过 `.slice()` 和 `.map()` 创建 **16 个临时数组**。在八个子步长、四次导数求值、60 帧每秒的条件下，这相当于 **每秒 1,920 次临时数组分配**，导致垃圾回收周期性引发 UI 卡顿。

解决方案为每个摆预分配 17 个 `Float64Array` 缓冲区（总计约 1.3 KB），每帧在所有 RK4 子步长中重复使用。**热路径零分配。** JIT 编译器识别出固定形状的类型数组，将其编译为高效的原生循环，彻底消除了 GC 干扰。

`p.ls_f64` Float64Array 镜像在四个变更点（`addJoint`、`removeJoint`、`rebuildChain`、`resizeCanvas`）与动态 `p.ls` 数组保持同步。

### 2.2 FPS Guardian——自调优应用

实现了一个滚动窗口帧率监视器（跟踪 30 个 `performance.now()` 时间戳），根据性能波动动态调整轨迹长度：

- **FPS < 55**：轨迹长度减少 50（下限：100）
- **FPS ≥ 58**：轨迹长度增加 10（上限：MAX_TRAIL_LENGTH）
- 调整操作每 30 帧节流一次，防止振荡

`MAX_TRAIL_LENGTH`（用户或视口定义的目标值）与 `TRAIL_LENGTH`（实时自适应值）解耦，确保 `resizeCanvas` 操作不会与 Guardian 的调整发生冲突。这构成了一个**渲染质量的闭环反馈系统**——在 Canvas 应用中并不常见的设计模式。

### 2.3 Path2D 批量渲染

先前的轨迹渲染实现每段调用 `beginPath()` + `moveTo()` + `lineTo()` + `stroke()`，每段约产生五次 JavaScript 到原生层的桥接开销。优化后的实现为每个不透明度批次构建单个 `Path2D` 对象，仅调用一次 `stroke()`，无论批次内有多少段，桥接开销均降至约两次。

### 2.4 带透明度动画的拖拽 HUD

拖拽摆锤时，出现极坐标网格叠加层：虚线圆显示摆锤的轨迹弧，24 条 15° 间隔的径向线提供吸附参考，亮白点标记吸附位置。叠加层约 125 ms 淡入、约 33 ms 淡出。`globalAlpha` 逐帧动画化——方法简单但有效，且避免了 CSS 动画的开销。

### 2.5 NaN 安全网与自动重置

数值爆炸是极端角度下混沌系统的固有风险。本实现采用**检测-恢复**模式：

```js
function isPendulumInvalid(p) {
    // 验证 thetas、omegas 和位置中是否存在 NaN/Infinity
    // 每帧 O(N)——计算开销可忽略
}
```

检测到 NaN 时，`safeResetPendulum()` 将速度归零、角度重置为 135°、清除轨迹。下一帧摆以安全状态重新出现。用户感知到的是短暂的视觉重置，而非系统崩溃。

该机制辅以硬上限（`MAX_LINKS = 8`、`MAX_PENDULUMS = 8`），防止用户进入质量矩阵的近奇异区域。

### 2.6 状态序列化与 URL 分享

完整的模拟状态被序列化为紧凑的 JSON（使用单字母键名以缩短 URL），编码为 URL 安全的 Base64，存储在 `window.location.hash` 中。恢复时用 `isFinite()` 和范围钳制验证每个字段，数据损坏时优雅降级。"分享"按钮将 URL 复制到剪贴板并附带动画反馈——**不需要后端、不需要数据库、不需要外部服务**。

`saveStateToURL()` 接入了 10 个变更点，确保任何交互后 URL 均可分享。

### 2.7 应用内 Wiki 阅读器

指南弹窗从简单的帮助页面演化为递归 Markdown Wiki 阅读器：
- 点击渲染后的 Markdown 中的 `.md` 链接被拦截，通过 `fetch()` 在应用内加载。
- 返回导航栏根据导航深度显示"← 返回操作说明"或"← 返回指南首页"。
- 支持任意深度导航（如 `README.md → AI_DISCLOSURE.md → AI_DISCLOSURE_ZH.md`）。
- `.md` 链接在应用内打开；外部链接以 `target="_blank"` 打开。

该架构使用户无需离开可视化器即可浏览项目文档。

### 2.8 可点击角度显示

左上角的角度显示从静态文本转化为**完全交互式的摆选择器**：

- 每个摆渲染为带 `data-idx` 属性的可点击 `<div>`。
- 点击选择摆，同时更新选择环、上下文菜单和指标图表。
- 使用 `pointerdown` 事件委托而非 `click`，避免图表画布处理程序中 `stopPropagation()` 的干扰。
- 短暂的背景闪烁提供选择的视觉确认。

该设计提供了一种无需键盘和触摸的摆切换机制，在移动设备上或指标面板打开时尤为实用。

### 2.9 每摆独立指标缓冲区

每个摆维护独立的 300 点环形缓冲区存储相空间和能量数据。`collectMetrics()` 独立遍历所有可见的摆。图表跟踪的摆取决于当前选择（无选择时默认使用 `pendulums[0]`）。因此：

- 切换选择**无缝**更新两个图表。
- 删除摆自动丢弃其关联数据。
- N 节摆使用**通用**的 N 摆拉格朗日能量公式。

### 2.10 移动端加固

- `{ passive: false }` 配合条件性 `preventDefault()`，防止拖拽时页面弹跳。
- 画布上设置 `touch-action: none`，指示合成器永不拦截触摸手势。
- 根元素设置 `overscroll-behavior: none`，抑制弹性过度滚动。
- 可折叠设置面板（齿轮图标）节省屏幕空间。
- 移动端轨迹长度减半至 600。
- 所有响应式调整集中于单个 `@media (max-width: 768px)` 块内。

## 3. 设计理念

### 3.1 渐进式复杂度

应用面向不同专业水平的用户，且不施加模式限制：

- **随意浏览**：打开应用，观察轨迹，按 Space 暂停。
- **产生好奇**：拖拽摆锤，开关混沌模式，观察蝴蝶效应。
- **主动探索**：添加关节构建 N 节链，生成多个摆。
- **深入分析**：打开指标面板，研究相图与能量守恒。
- **艺术创作**：配置颜色、切换摆的可见性、导出高清 PNG 作品。
- **高级用户**：通过 URL 分享配置，使用键盘快捷键操作一切。

不施加模式锁定；每个功能均可从基础状态直接发现。

### 3.2 零外部依赖

整个系统——物理引擎、渲染引擎、UI 框架、绘图库、帮助系统、状态序列化——全部使用**原生 HTML、CSS 和 JavaScript** 构建。没有框架、没有构建工具、没有 `package.json`。这带来了若干实际优势：

- **瞬间加载**：没有 CDN 请求、没有打包解析、没有框架初始化开销。
- **随处可用**：离线可用、`file://` 协议可用、任何 HTTP 服务器均可运行。
- **长久可用**：免疫框架更替——十年后这段代码仍可在浏览器中运行。
- **性能优异**：没有抽象层开销、没有虚拟 DOM、没有响应式系统税。

### 3.3 文档即架构

英文与中文两套并行设计记录，完整记录了开发过程中的**每个决策、每个缺陷、每个修复**。这一做法将开发叙事视为项目制品的有机组成部分。
