# 设计记录 — 第一阶段

> **语言**：中文 · [English](THOUGHTS.md)

## 项目
**混沌艺术 — 双摆可视化器**  
一个高性能、极简主义的 HTML5 Canvas 网页应用，专注于双摆的美学流动与混沌分叉。

## 架构

### 文件结构
| 文件 | 用途 |
|------|------|
| `index.html` | 极简 HTML 外壳，含 HiDPI 视口 meta 标签 |
| `style.css` | 全屏暗色主题（`#0a0a0f`），无滚动条 |
| `script.js` | 物理引擎、画布设置、渲染、动画循环 |
| `Plan.md` | 原始项目计划（包含所有 5 个阶段） |

### 物理引擎（RK4 积分器）
- **一致的单位制**：模拟使用 `L = 1.5 m`，`g = 9.81 m/s²`，`m = 10 kg`。`pxPerUnit` 缩放因子在窗口大小变化时重新计算，将物理单位映射为画布像素。
- **带 4 个子步长的 RK4**（`h = 1/240 s`）防止能量漂移——对于欧拉积分会爆炸的混沌系统至关重要。
- **固定时间步长**（`dt = 1/60`）使物理速度与帧率无关。

双摆方程遵循标准公式（`derivatives()` 函数中实现导数计算），RK4 步进在 `rk4Step()` 中实现。

### 画布与 HiDPI
- `canvas.width/height = CSS 尺寸 × devicePixelRatio`
- `ctx.setTransform(dpr, 0, 0, dpr, 0, 0)` 保持所有绘图在 CSS 像素坐标系中进行
- 窗口大小变化监听器重新计算缩放因子、固定点位置和摆锤位置

### 摆的状态
存储于单一 `state` 对象中：
- 角度位置 + 速度（物理量）
- 像素位置（通过角度 × 缩放因子计算）
- 两个轨迹数组（每个摆锤一个，各上限 1200 点）

## 第一阶段改进（回顾）
- 从半透明叠加淡出（会导致摆体结构的残影）改为采用梯度不透明度的批处理线段轨迹
- 为摆锤 1 的轨迹添加了与其颜色匹配的着色

# 设计记录 — 第二阶段

## 方法
原计划建议在轨迹画布上使用半透明叠加实现"累积淡出"。在第一阶段开发中，用户更倾向于**逐渐变淡的线段**而非模糊的残影发光效果。第二阶段据此调整为双层架构。

## 双层画布架构

### 层 A（底层）— 轨迹画布（`#canvas-a`）
- **用途**：显示两个摆锤的渐隐轨迹。
- **渲染策略**：每帧 `clearRect` 清除整个图层，然后将所有存储的轨迹点重新绘制为连接线段，分为 80 个不透明度批次。最旧的线段 `α ≈ 0.02`，最新的 `α ≈ 0.90`。
- **为何用清除+重绘而非累积叠加**：防止任何残影/模糊。每帧都是已知轨迹数据的干净渲染。
- **画布可见性**：透明背景 — 未绘制区域透出 body 的 `#0a0a0f`。

### 层 B（顶层）— 摆体画布（`#canvas-b`）
- **用途**：显示摆杆、摆锤和固定点。
- **渲染策略**：每帧 `clearRect`，然后以完全不透明度绘制所有摆体元素。
- **堆叠顺序**：`z-index: 1` 使其位于层 A 之上。

### 为何使用双层？
1. **关注点分离**：轨迹渲染（开销大、数据驱动）与摆体渲染（开销小、几何驱动）隔离。
2. **可扩展性**：混沌模式（第三阶段）将添加第二个摆——其轨迹可干净地叠加在层 A 上。导出/保存作品（第五阶段）可合并两个图层。控制覆盖层（第四阶段）可放在第三层上。
3. **性能**：轨迹渲染不影响摆体渲染。每个图层都有自己简化的合成模型。

## 关键决策：为何不用累积叠加？
原始第二阶段计划要求在轨迹层上使用 `rgba(0,0,0,0.02)` 叠加。用户明确拒绝了这种视觉效果（残影/模糊），转而选择**具有梯度不透明度的清晰线段**。批处理不透明度方法提供了：
- **清晰的线条**：每段都是清晰的 1.5px 描边。
- **可控的淡出**：不透明度是确定性的——从尾部到末端严格线性变化。
- **无残影**：因为轨迹画布被完全清除并重绘，不会有陈旧像素的累积。

## 验收标准状态
- ✅ **平滑渐隐轨迹**：线条在大约 1200 个点内从几乎不可见过渡到明亮。
- ✅ **清晰的摆体**：在其独立的清除图层上——无残影。
- ✅ **稳定的 60fps**：两层画布 + 简单合成——即使在长时间运行后也无性能问题。

# 设计记录 — 第三阶段

## 方法
从单一的 `state` 对象重构为 `pendulums[]` 数组，使得摆 B 可以在运行时以微小的角度偏移生成，展示混沌分叉。

## 架构变化

### 从单一状态 → 摆数组
```
之前：const state = { theta1, theta2, omega1, omega2, bob1X, …, trail1, trail2 }
之后：const pendulums = [ { theta1, …, trail1, trail2, color1, color2 }, … ]
```
- 共享量（固定点 `PIVOT`、`pxPerUnit`）保存在数组外部。
- `createPendulum(theta1, theta2, color1, color2, copyTrailsFrom?)` 是工厂函数。
- `copyTrailsFrom` 参数使得摆 B 可以继承摆 A 的轨迹历史，使两条轨迹看起来从同一起点分叉。

### 混沌模式的工作原理
1. **切换（`C`键）**：`toggleChaos()` 添加或移除第二个摆。
2. **生成**：摆 B 读取摆 A 的当前角度，加上 `CHAOS_OFFSET`（0.01° ≈ 0.000175 rad），并复制 A 的轨迹数组。
3. **物理循环**：`stepPhysics` 遍历 `pendulums`，使用相同的 RK4 子步长——数值处理方式完全相同，仅初始条件不同。
4. **渲染循环**：`draw()` 遍历 `pendulums`，在层 A 上绘制轨迹，在层 B 上绘制摆体。摆 B 在层 B 上绘制于摆 A 之上。

### 配色方案
| 元素 | 摆 A | 摆 B |
|------|------|------|
| 摆锤 1（上部） | `#6080c0` 蓝色 | `#c060a0` 品红 |
| 摆锤 2（下部/末端） | `#00d4ff` 青色 | `#ff60c0` 粉色 |
| 摆杆 | `#404060` | `#604060` |

### 为何在当前位置生成而非初始角度
原计划说"摆 B 在摆 A 的**初始**角度上加 0.01° 生成。"在**当前**角度生成能提供更好的演示效果：
1. 用户可在任意时刻切换混沌模式并立即看到分叉。
2. 轨迹在切换点可见地分叉（摆 B 继承 A 的轨迹，随后分道扬镳）。
3. 展示混沌无处不在——任何微小的初始差异，在任何时刻，都会爆炸式放大。

## 验收标准状态
- ✅ **两个摆无性能下降**：4 条轨迹 × 1200 点 = 4800 点，160 批绘制调用，2 次摆体绘制调用——远在 60fps 预算内。
- ✅ **分叉点可见**：摆 B 继承 A 的轨迹 → 两者始于同一原点，然后 `drawTrail` 独立绘制 → 青色和品红色轨迹可见地分叉。

# 设计记录 — 控制（播放/暂停与重置）

这些是在第三阶段和第四阶段之间临时添加的，因为用户在生产完整的 UI 覆盖层之前就需要模拟控制。

## 播放/暂停（Space）
- **暂停**完全冻结模拟：所有角度、角速度和轨迹缓冲区原样保留。
- 动画循环继续运行，但当 `paused = true` 时跳过 `stepPhysics()`。
- 渲染继续，显示冻结的摆体和所有累积的轨迹。
- 取消暂停后从精确保存的状态恢复物理计算——无动量损失，无跳变。

## 重置（R）
- 将摆 A 重置为初始角度（θ₁ = θ₂ = 0.75π，ω₁ = ω₂ = 0）。
- 清空摆 A 的轨迹数组 → 下一帧轨迹画布重新绘制时为空白。
- 如果混沌模式激活，移除摆 B 并将 `chaosMode` 设为 `false`。
- 暂停状态**不会**因重置而改变——用户可以在暂停时重置以进行拖拽设置。

## 统一标签
一个统一的 `updateCaption()` 函数驱动屏幕上的文字，保持 DRY：

```
[Space] 暂停  ·  [R] 重置  ·  [C] 混沌
```

切换时标签动态更新：暂停 ↔ 播放，混沌 ↔ 单摆。

# 设计记录 — 第四阶段

## 方法
第四阶段增加了两大功能：**可点击的叠加按钮**（取代纯文本标签）和**拖拽设置**，用于直观地操控初始条件。

## 叠加控制栏

### 设计
- 半透明药丸条居中固定在底部：`background: rgba(10,10,15,0.6)`，`border-radius: 20px`，`backdrop-filter: blur(4px)`。
- 四个 `<button>` 元素以 `gap: 12px` 的 flex 行布局。
- 文字按钮起始为 `rgba(255,255,255,0.35)`，悬停时提亮至 `0.7`。
- 这些按钮是可交互的 HTML 元素（而非画布绘制），因此具有可访问性并遵循平台约定。

### 为何用按钮取代纯文本标签？
用户最初只想要"一行字"。第四阶段计划明确要求可点击的控制项。按钮比键盘快捷键更易于发现——新用户看到"⏸ 暂停"就知道可以点击。键盘快捷键（Space、R、C）对高级用户仍然有效。

### 清除轨迹
`clearTrails()` 清空所有激活摆的轨迹数组。由于轨迹画布每帧都会 `clearRect` 并从头重绘，清空数组就足够了——下一帧自然显示空轨迹。这与重置不同：摆保持当前位置和速度。

## 拖拽设置

### 工作原理
1. 用户按 Space（或点击 ⏸ 暂停）冻结模拟。
2. 用户将鼠标悬停在摆锤上 → 光标变为 `grab`。
3. 用户点击摆锤 → 光标变为 `grabbing`，拖拽开始。
4. 用户移动鼠标 → 摆锤跟随光标，约束在其杆长范围内。
   - **拖拽摆锤 1**：`θ₁ = atan2(mouseX - PIVOT.x, mouseY - PIVOT.y)`
   - **拖拽摆锤 2**：`θ₂ = atan2(mouseX - bob1X, mouseY - bob1Y)`
5. 拖拽期间两个角速度均设为 0（静态初始条件）。
6. 用户松开 → 摆锤保持在新角度。

### 为何用 `atan2(dx, dy)`？
在画布坐标系中，`y` 轴向下递增，摆的静止位置（θ=0）指向正下方。`atan2(dx, dy)` 在光标位于参考点正下方时返回 0，光标偏右时为正，偏左时为负——与摆的角度约定完全一致。

### 触控支持
`touchstart`/`touchmove`/`touchend` 处理程序与鼠标逻辑完全相同，使用 `passive: true`，因为全屏画布不存在滚动干扰。

### 混沌模式交互
拖拽仅影响**摆 A**。摆 B 保持自身状态。如果用户在拖拽后关闭再打开混沌模式，摆 B 将在 A 的新角度 + 偏移处重新生成——从拖拽点开始产生新的分叉。

## 验收标准状态
- ✅ **通过拖拽设置状态**：暂停时可用。摆锤吸附到光标位置。速度归零。
- ✅ **UI 占用 < 10% 屏幕**：底部细长的药丸条。控制栏约 35px 高，远小于视口的 10%。

# 设计记录 — 第五阶段

## 方法
以最少的代码更改实现了所有三个第五阶段功能，尽可能复用现有基础设施。

## 速度自适应线宽

### 工作原理
- 每个 `trail2` 点现在存储 `{x, y, s}`，其中 `s` = 自上一帧以来的像素距离（瞬时速度的代理量）。
- 在 `drawTrail` 中，当 `velocityStyle = true` 时，每批次的平均 `s` 被计算并映射到 `lineWidth`：
  - `avg = 0`（静止）→ `lineW = 3.0`（粗、深色）
  - `avg ≥ 150`（快速）→ `lineW = 0.8`（细、亮色）
  - 在上述极值之间线性插值。
- 摆锤 1 的轨迹（`trail1`）**不**使用速度样式——计划明确提到"第二个摆锤的速度"。

### 为何用像素距离而非角速度？
`Math.hypot(Δx, Δy)` 在位置计算中已经可用，零额外开销，且直接衡量屏幕上的视觉速度。使用角速度需要额外公式，且无法解释依赖于几何结构的杠杆臂放大效应。

## 慢动作模式

### 实现
- 单一布尔值 `slowMo` 切换物理 dt 在 `1/60` 和 `1/120` 之间。
- 渲染循环通过 `requestAnimationFrame` 保持 60fps。
- 切换后，摆体以半速运动，让观众追踪复杂的混沌运动。
- 子步长参数不变——`SUB_STEPS = 4` 保持恒定，因此每个子步长在正常模式下为 `dt/4 = 1/240`。慢动作下更小的子步长实际上改善了能量守恒。

## 导出作品

### 实现
一个 10 行的函数：
1. 创建一个离屏 `<canvas>`，分辨率与显示器的 HiDPI 分辨率相同。
2. 依次绘制 `canvasA`（轨迹）和 `canvasB`（摆体）——顺序与视觉堆叠一致。
3. 通过临时 `<a>` 元素触发浏览器下载，`download="double-pendulum-art.png"`，`href` 设为 `canvas.toDataURL('image/png')`。

导出的图片包含完整的累积轨迹和导出瞬间的摆体——是作品的真实验证快照。

## 控制栏更新
药丸条现在包含 6 个按钮。`gap: 12px` 和 `font-size: 12px` 使控制栏在桌面和平板视口上保持紧凑。

## 验收标准状态
- ✅ **艺术深度**：轨迹粗细随摆速有机变化。
- ✅ **下载**：一键保存 HiDPI 合并 PNG，包含两个图层。

## 所有阶段完成 🎉
1. ✅ 第一阶段 — 物理引擎与 HiDPI 基础
2. ✅ 第二阶段 — 双层画布与轨迹美学
3. ✅ 第三阶段 — 混沌模式与状态架构
4. ✅ 第四阶段 — 极简控制与直接操控
5. ✅ 第五阶段 — 视觉润色与导出

# 第六阶段 — 多摆沙盒与个性化

# 界面优化 — 角度显示与角度吸附

在第六阶段和第七阶段之间作为独立改进添加。

## 角度显示
屏幕顶部居中实时显示：`θ₁ xx.x°  θ₂ xx.x°`。跟踪选中的摆（未选择时使用摆 A）。每帧在 `animate()` 中更新。

对于 N 节摆（第七阶段），显示动态使用下标字符显示 N 个条目。

## 角度吸附
在暂停状态下拖拽摆锤时，如果角度在 5° 以内，角度会吸附到最近的 15° 倍数。这模仿了 GeoGebra 和 3D 建模工具中的磁吸对齐功能。

- `SNAP_DEG = 15` — 吸附网格间距
- `SNAP_THRESHOLD = 5` — 激活窗口（度）
- 同时应用于摆锤 1 和摆锤 2，鼠标和触控拖拽均适用——通过 `snapAngle(rad)` 包装函数实现

# 第七阶段 — 基于 Verlet 积分的 N 节摆

## 目标
使用 Verlet 积分 + 距离约束替换 RK4 角度物理，实现动态 N 节链。

## 实现

### Verlet 物理引擎
每个摆存储：
- `particles: [{x, y, px, py}, ...]` — 当前位置和上一帧位置（通过 `v = x - px` 隐式表示速度）
- `constraints: [{a, b, len}, ...]` — 成对距离约束

每帧：
1. 对所有粒子施加重力（固定点 0 除外）：`y += vy + G * pxPerUnit * VERLET_G_SCALE * h²`
2. 运行 `max(10, constraints * 2)` 次约束求解器迭代以强制杆长。
3. 将固定点约束到 `PIVOT`。
4. 为每个粒子记录轨迹点（每个粒子一个轨迹数组）。

### 被移除的内容
- `derivatives()` 和 `rk4Step()` — 被 `verletStep()` 替换
- `computeBobPositions()` — 被 `syncBobPositions()` 替换
- `theta1`、`theta2`、`omega1`、`omega2` — 被粒子位置替换
- 双轨迹系统（`trail1`/`trail2`） — 被每粒子 `trails[]` 数组替换

### 保留的内容
- `bob1X/Y` 和 `bob2X/Y` 作为向后兼容的计算字段
- 所有第六阶段功能（选择、可见性、颜色循环、上下文菜单、删除）
- 混沌模式、慢动作、保存作品、清除轨迹、角度显示、角度吸附
- 键盘快捷键、拖拽设置、触控支持

### 每粒子轨迹
每个粒子拥有自己的 `trails[i]` 数组。轨迹长度随深度缩放：
- **粒子 1**（第一个摆锤）：约 180 点（3 秒），以固定 1.5px 线宽绘制，使用 `color1`
- **中间粒子**：轨迹逐渐增长至末端
- **末端粒子**：完整 1200 点（20 秒），速度自适应线宽（0.8–3.0 px），使用 `color2` 绘制

轨迹长度公式：`limit = TRAIL_LENGTH × (0.15 + 0.85 × i / (N-1))`

### 关节修改器
- `➕` 沿最后一个线段的方向延长链，长度 = 上一个连接 × 0.85。添加一个粒子和一个约束。压入一个空轨迹数组。
- `➖` 移除最外层的连接（最少 2 个）。弹出粒子、约束和轨迹数组。
- 选中摆时在上下文菜单中显示按钮；链接数达到 `MIN_LINKS` 时隐藏 `➖`。

### 美学缩放
每个新连接是上一个连接的 85%。摆锤半径类似缩放：`r × LINK_SCALE^(i-1)`。内部摆锤使用 `color1`；末端摆锤使用 `color2`。

### 角度显示更新
对于 N 节摆，显示 N 个条目：`θ₁ xx° θ₂ xx° … θₙ xx°`，通过每个线段的方向角 `atan2(dx, dy)` 计算。

## 挑战与修复

1. **重力太弱**：Verlet 重力 `G × pxPerUnit` 仅产生约 0.022 像素/子步长的位移——摆体看似冻结。修复：`VERLET_G_SCALE = 8` 将重力放大以匹配 RK4 时代的摆动速度。

2. **单一轨迹丢失内部摆锤**：只有末端留下轨迹；添加关节后所有内部摆锤在轨迹中不可见。修复：每粒子 `trails[]` 数组，按深度缩放长度。

3. **Verlet ↔ 角度模型转换**：调整大小、重置需要根据目标角度重建链。`buildChain(nLinks, thetaDeg)` 在指定角度创建粒子。`rebuildChain()` 被 `resetSimulation` 使用。

4. **Verlet 下的混沌偏移**：0.01° 角度偏移应用为末端粒子 0.3px 的垂直位移——小到足以产生蝴蝶效应分叉。

5. **向后兼容**：`bob1X/Y` 和 `bob2X/Y` 每帧通过 `syncBobPositions()` 从粒子同步，使选择圈、命中检测和角度显示无需更改即可工作。

6. **关节更改后的陈旧轨迹**：`addJoint` 和 `removeJoint` 现在压入/弹出轨迹数组，保持轨迹数量与粒子数量同步。窗口调整大小时清除轨迹。

# 缺陷修复 — 重置所有摆与拖拽稳定性

## 缺陷 1：重置只触及 pendulums[0]

`resetSimulation()` 只遍历 `pendulums[0]`，其他额外添加的摆（通过 `+`）保持当前角度和轨迹不变。

**修复**：将 `resetSimulation()` 改为遍历数组中的每个摆，在 `DEFAULT_ANGLE_DEG (135°)` 重建每条链，同步摆锤位置，并清空所有轨迹数组。混沌模式仍然移除摆 B 并将 `chaosMode` 重置为 false。

## 缺陷 2：拖拽末端应围绕内球旋转，而非固定点

最初的尝试以支点到鼠标的角度重建整条链（从固定点的共线旋转）。用户拒绝了这一点：*"我的意思是移动外球时，希望它可以以里层球为圆心做圆弧运动"*。

**根本原因**：约束求解器的 50/50 修正分配移动了内球，因为它是吸收两个相邻约束修正的唯一自由端。但即使是固定点旋转修复也是错误的——末端应围绕其父节点（内球）摆动，保持父节点固定。

**最终修复 — 圆约束拖拽**：

拖拽**任意**粒子时：

1. **约束到父节点的圆**：计算从父粒子（partIdx − 1）到鼠标的角度。执行吸附（`snapAngle`）。将拖拽的粒子放置在距父节点 `rodLen` 的方向上。父节点完全固定——只有这个线段的角度改变。

2. **前向传播**：对于从 `partIdx` 到末端的每个约束，将每个子粒子精确放置在其父粒子的 `constraint.len` 距离处，保持方向不变。这使得所有下游连接保持完整，而不移动拖拽点上游的任何粒子。

这种方法适用于 N 节链中的任意粒子，并自然支持正在调整的线段的角度吸附。

# 第八阶段 — 精确拉格朗日 RK4 物理引擎

## 动机

用户报告：*"双摆的轨迹看起来太规律了，缺乏混沌运动，即使在初始大角度下也是如此"*——并特别询问是否存在小角度近似或过大的阻尼。

## 根本原因

第七阶段的 Verlet + 约束求解器方法存在**严重的数值阻尼**：

1. **每帧 40 次约束修正**：4 个子步长 × 10 次约束迭代 = 每帧 40 次位置调整，每次应用 50/50 的修正分割。这起到了隐式阻尼的作用——重复的修正每帧都会消耗动能。

2. **无非线性耦合**：Verlet 独立处理每个粒子（重力 + 距离约束）。没有编码两臂之间科里奥利类耦合的项（`ω₁² sin(θ₁-θ₂)` 等）。没有这些项，驱动混沌的初始条件敏感依赖性就不存在。

3. **均匀的 50/50 分割**：约束求解器在约束的两个粒子之间均匀分配修正。对于链式摆，这意味着内球每次迭代都需要吸收**两个**相邻约束的一半修正——实际上使其"摩擦"加倍。

结果是：摆像一个阻尼振荡器一样摆动，趋于周期性运动，而非表现出真正的混沌分叉。

## 解决方案：拉格朗日力学 + RK4

### 精确运动方程

双摆拉格朗日函数（等质量 m 在杆末端，长度 l₁、l₂，角度 θ₁、θ₂，从垂直向下方向测量）：

```
L = T - V
  = ½m[2l₁²ω₁² + l₂²ω₂² + 2l₁l₂ω₁ω₂ cos(θ₁-θ₂)]
    + mg[2l₁ cos θ₁ + l₂ cos θ₂]
```

应用欧拉-拉格朗日方程得到角加速度的 **2×2 线性系统**：

```
2 l₁ α₁  +  l₂ cos(Δ) α₂  =  -2g sin θ₁  -  l₂ ω₂² sin(Δ)
 l₁ cos(Δ) α₁  +  l₂ α₂  =  -g sin θ₂   +  l₁ ω₁² sin(Δ)
```

其中 Δ = θ₁ − θ₂。

此方程组直接求解（克莱姆法则）——**无小角度近似**，完整的三角耦合保留。关键项：

- `sin(Δ)` 耦合在摆臂未对齐时放大分叉
- `ω₁² l₁ sin(Δ)` 和 `ω₂² l₂ sin(Δ)` — 科里奥利类项产生敏感依赖性
- `cos²(Δ)` 分母 → 特定摆臂配置下的非线性共振
- 行列式 = `l₁ l₂ (2 − cos²(Δ)) ≥ l₁ l₂ > 0` — 永不为零，无奇点

### RK4 积分

每帧 4 个子步长（h = 1/240 s），对 4 维状态（θ₁、θ₂、ω₁、ω₂）进行积分。RK4 方法的四阶精度在长时间内保持能量——积分器本身不会引入人工阻尼。

### 架构变化

| 之前（Verlet） | 之后（拉格朗日 RK4） |
|---------------|---------------------|
| `particles[]` 含 `{x, y, px, py}`（隐式速度） | `particles[]` 仅含 `{x, y}` |
| `constraints[]` 用于物理计算（距离求解） | `constraints[]` 仅用于存储长度 |
| `verletStep()` — 每帧 40 次约束迭代 | `derivatives()` + `rk4Step()` — 精确 2×2 求解 |
| `VERLET_G_SCALE = 8` 伪造摆动速度 | `G = 9.81` 直接使用（正确物理） |
| `CONSTRAINT_ITERS = 10` | 完全移除 |
| N 节通过独立粒子链实现 | N 节通过沿杆 2 的比例插值实现 |

### N 节处理

对于 N > 2，初始实现将中间粒子**沿杆 2 按比例放置**（与 θ₂ 方向共线），使用仅有 2 个自由度的 2×2 拉格朗日求解器。这导致**第 3 个摆锤被刚性地锁定在第 2 个摆锤上**——它们共享相同的 θ₂，关节处无独立弯曲。用户报告为：*"最后两个摆锤完全锁定在一起，像一个刚体一样运动。"*

**修复**：将硬编码的 2×2 求解器替换为通用的 N 摆拉格朗日求解器。状态存储数组 `thetas[]`、`omegas[]`、`ls[]`。`derivativesArray()` 构建 N×N 质量矩阵：

```
Mᵢⱼ = lᵢ lⱼ (N−max(i,j)) cos(θᵢ−θⱼ)
bᵢ  = −Σⱼ≠ᵢ Aᵢⱼ sin(θᵢ−θⱼ) ωⱼ² − g lᵢ (N−i) sin(θᵢ)
```

通过带部分主元的高斯消元法求解 `M·α = b`——适用于任意 N。`computeParticlePositions` 使用简单的前向循环：每个连接都有自己独立的角度。`dragParticle` 设置 `thetas[partIdx-1]`，针对任意 N 都能正确处理对应的连接。

### 右键点击缺陷

用户报告：*"在屏幕上右键点击时，底部的控制图标（调色板、加号按钮等）消失了。"*

## 遇到的缺陷与修复

### 缺陷 1：能量图无读数（空白画布）

**根本原因** — 两个独立的缺陷：

1. **`(_, i) => i` 索引未传递**。`drawSolidLine` 以单参数调用 `getX(data[i])`，但 getter `(_, i) => i` 期望第二个参数为索引。缺少第二个参数时，`i` 为 `undefined`，`mapX(undefined, ...)` 对每个 X 坐标产生 `NaN`。路径永远无法绘制。

   ```
   之前：const px = mapX(getX(data[i]), xMin, xMax, PLOT_W);
   之后：const px = mapX(getX(data[i], i), xMin, xMax, PLOT_W);
   ```

   同样的修复应用于 `drawFadingLine`（相图的 getter `d => d.thetas[0]` 忽略额外参数，因此无害）。

2. **`const` 赋值给 `PLOT_W`/`PLOT_H`**。原始代码声明了 `const PLOT_W = 290; const PLOT_H = 180;`。`setupPlotCanvas` 需要将它们重新赋值为 CSS 计算尺寸（`canvas.clientWidth`/`clientHeight`）以适应全屏布局。改为 `let`。

### 缺陷 2：总能不守恒（341% "漂移"）

初始实现后，`totalE` 在 -29 到 +113 之间剧烈波动——远非验收标准要求的平直线。

**根本原因**：势能公式使用了 y 轴向下画布惯例（`PE = g Σ (N−i) lᵢ cos(θᵢ)`），但物理引擎的拉格朗日函数使用 y 轴向上惯例。在运动方程中，重力项为 `−g (N−i) lᵢ sin(θᵢ)`（参见 `derivativesArray` 第 237 行），这与 E-L 方程仅在 `V = −g Σ (N−i) lᵢ cos(θᵢ)` 时才一致。

**修复**：对 PE 公式取反：
```
之前：PE =  g Σ (N−i) lᵢ cos(θᵢ)   →  totalE 漂移
之后：PE = −g Σ (N−i) lᵢ cos(θᵢ)   →  totalE 恒定至 0.0000%
```

验证：initialE = 29.6545，finalE = 29.6545，漂移 = -0.0000%（300 步后）。

### 缺陷 3：侧面板太小，图表无法阅读

初始的右侧停靠面板（290px 宽）使图表过小无法解读。用户请求全屏展示。

**修复**：将侧面板布局替换为全屏 flexbox 覆盖层。图表画布由 CSS 调整大小（`flex: 1` 在 flex 列内），JavaScript 每帧读取 `canvas.clientWidth`/`clientHeight`，以 `devicePixelRatio` 分辨率设置实际画布像素尺寸。图表现在填满可用视口区域。

### 缺陷 4：M 键不易发现

初始实现只绑定了 `M` 键，但没有可见提示。

**修复**：在底部控制栏添加 `📊 Metrics [M]` 按钮，与其他操作的约定一致（例如 `⏸ Pause [Space]`、`⚡ Chaos [C]`）。按钮点击调用 `toggleMetricsPanel()`，键盘快捷键独立工作。

### 缺陷 5：全局指标数组 — 忽略多摆与 N 节

**问题**：指标系统使用单一的全局 `metricsData` 数组。
- 有多个摆时，切换选择或生成新摆会导致图表显示过时/错误的数据（数组属于收集时恰好为 `selectedPendulum` 的摆）。
- 相图标题硬编码为 `"Phase Space — θ₁ vs ω₁"`，没有指出显示的是哪个摆。

**根本原因**：`metricsData` 是一个独立数组，由 `collectMetrics()` 推入数据，该函数每帧只跟踪一个摆：

```js
// 旧代码 — 单一全局缓冲区，仅跟踪一个摆
let metricsData = [];
function collectMetrics() {
    const p = selectedPendulum ?? pendulums[0];
    if (!p || !p.visible) return;
    metricsData.push({ ... });  // ⟶ 每帧为同一个摆覆盖写入
}
```

**修复**：每个摆拥有自己的 `metrics[]` 数组，在 `createPendulum()` 中初始化。`collectMetrics()` 现在遍历**所有**可见摆：

```js
// 新代码 — 每摆独立缓冲区
function collectMetrics() {
    for (const p of pendulums) {
        if (!p.visible) continue;
        p.metrics.push({ thetas, omegas, KE, PE, totalE });
        while (p.metrics.length > METRICS_CAPACITY) p.metrics.shift();
    }
}
```

`getTrackedPendulum()` 辅助函数返回图表渲染器应使用的摆（选中的或 `pendulums[0]`）。相图标题每帧动态更新：`"Phase Space — Pendulum {id}  θ₁ vs ω₁"`。

# 第八阶段改进 — 内边距与刻度标签

## 动机

绘图区域没有内部边距，导致：
1. X 轴标题（`θ₁(°)` / `step`）与坐标数字重叠
2. 左侧文本（轴标签、刻度数字）被画布边缘裁剪
3. 图表标题和图例挤在左上角，彼此重叠

此外，Y 轴标签显示丑陋的任意小数，如 `62.87` 和 `-53.21`，因为默认使用 `.toFixed(2)`。

## 解决方案：`PLOT_MARGIN`

边距对象定义了内部绘图区域周围的缓冲区：

```js
const PLOT_MARGIN = { left: 58, right: 58, top: 38, bottom: 40 };
```

内部绘图区域是数据绘制的地方：

```js
function innerW() { return PLOT_W - PLOT_MARGIN.left - PLOT_MARGIN.right; }
function innerH() { return PLOT_H - PLOT_MARGIN.top - PLOT_MARGIN.bottom; }
```

### 坐标映射

`mapX`/`mapY` 被 `pxX`/`pxY` 替换，将数据值映射到内部绘图区域内的像素位置：

```
pxX(val, min, max) = PLOT_MARGIN.left + innerW() * (val - min) / (max - min)
pxY(val, min, max) = PLOT_MARGIN.top  + innerH() * (1 - (val - min) / (max - min))
```

如果 `min === max`（退化范围），坐标居中在内部区域。

### 各元素的绘制位置

| 元素 | 区域 | 细节 |
|------|------|------|
| 网格线 | 内部区域 | `left…right`，`top…bottom` |
| 数据线 | 内部区域 | 裁剪到 `left ± 2`、`right ± 2` 等 |
| 刻度标记 | 内部区域紧外侧 | X 刻度延伸至 `bottom` 以下，Y 刻度延伸至 `right` 以右 |
| 刻度标签 | 边距区 | X 标签：`y = bottom + 7`，Y 标签：`x = right + 9` |
| 轴名称 | 边距区 | X 轴名称：右下角，Y 轴名称：上边距上方 |
| 图表边框 | 内部区域周围 | `strokeRect(left, top, innerW, innerH)` |
| 图例（能量） | 内部区域左上 | 锚定于 `(left + 6, top + 6)` |

### `textBaseline` 和 `textAlign` 的使用

每个标签块都显式设置这两个属性，避免画布状态污染：

- X 轴刻度标签：`textAlign: 'center'`、`textBaseline: 'top'`
- Y 轴刻度标签：`textAlign: 'left'`、`textBaseline: 'middle'`
- 轴名称：`textAlign: 'left'`、`textBaseline: 'alphabetic'` / `'top'`

## 浮点标签清理

- 所有 Y 轴格式字符串从 `.toFixed(2)` 改为 `.toFixed(1)`（最多 1 位小数）
- 相图 ω₁：`123.5` 而非 `123.45`
- 能量值：`29.7` 而非 `29.65`
- 添加 `niceNum()` 辅助函数（尚未用于刻度间隔吸附，但可用于未来改进）

# 第八阶段改进 — 固定的 0°–360° 范围与可点击摆选择器

## 固定相图 X 轴范围为 [0°, 360°]

之前，相图的 X 轴自动缩放到数据中出现的任何 θ₁ 范围（例如 −200° 到 500°）。这使得图表随着角度漂移而放大/缩小，隐藏了吸引子形状。

**修复**：绘图前将所有 θ₁ 值归一化到 [0, 2π)：
```js
const norm = v => ((v % twoPi) + twoPi) % twoPi;
const X = data.map(d => norm(d.thetas[0]));
```
X 范围固定为 [0, 2π]（0°–360°）。ω₁=0 线仍作为水平零轴绘制。左边缘作为 θ₁=0 线，因此不需要垂直零轴。

## 可点击角度显示用于摆选择

之前，切换摆需要暂停并点击画布上的摆锤。当分析面板打开时这很麻烦（需要关闭面板、暂停、点击、重新打开）。

**修复**：左上角的角度显示现在将每个摆渲染为可点击的 `<div>`，带有 `data-idx` 属性：
```html
<div class="pend-entry" data-idx="0" style="color:#00d4ff">
  <span class="marker">●</span> θ₁ 46.3°  θ₂ 80.6°
</div>
```
点击任意摆条目调用 `selectPendulum(idx)`，更新摆上的选择环，显示上下文菜单，并切换分析图表显示的数据。

### CSS 变更
- `pointer-events: none` 从 `#angle-display` 移除 → 条目现在可交互
- `.pend-entry` 增加 `cursor: pointer`，悬停时圆角背景（`rgba(255,255,255,0.08)`）
- `.pend-entry.sel` 获得略亮背景（`rgba(255,255,255,0.1)`）
- `.marker` 跨度固定 14px 宽度，保证 `▸` 和 `●` 对齐一致

### 缺陷修复：相图环绕时产生跨图线条

当 θ₁ 从约 359° 回绕到约 1°（跨越 0° 边界）时，归一化后的数据值位于 [0, 2π) 范围的两端，在整张图上产生一条直线。

**修复**：`drawFadingLine` 现在检测连续 X 数据值的跳跃是否超过范围的一半（wrapThreshold = range × 0.5）。检测到此类跳跃时，当前路径被描边并在当前点开始新路径——与越界裁剪逻辑完全相同。光晕和核心通道各自独立处理此情况。

### 缺陷修复：角度显示点击未到达处理程序

`.pend-entry` 点击处理程序使用 `document.getElementById('angle-display').addEventListener('click', ...)`。图表画布的 `click` 处理程序调用 `e.stopPropagation()`，可能阻止 `click` 事件到达文档。此外，放大时图表框位于 z-index 10，拦截了指向下面角度显示的所有指针事件。

**修复**：改用文档级 `pointerdown` 事件委托（与 `click` 不同的事件类型，不受 `click` 处理程序的 `stopPropagation()` 影响）：
```js
document.addEventListener('pointerdown', (e) => {
    const entry = e.target.closest('.pend-entry');
    if (!entry || !document.getElementById('angle-display').contains(entry)) return;
    // … 选择摆，闪烁背景
});
```
同时还添加了短暂的背景闪烁（`rgba(255,255,255,0.18)` → 下一帧移除）作为点击已注册的视觉确认。

### 缺陷修复：缓冲区填满后能量图 X 轴停滞

之前能量图使用数组索引作为 X 值（`(_, i) => i`），`xRange.max = data.length - 1`。300 点环形缓冲区填满后，`data.length` 永远保持在 300，因此 X 轴标签（0、60、120、180、240、299）永远不变——用户看不到时间在流逝。

**根本原因**：数组索引对于环形缓冲区不是单调递增的——它会回绕。

**修复**：添加全局 `globalMetricsStep` 计数器，每次 `collectMetrics()` 运行时递增（每物理帧一次）。每个指标条目存储其步进编号。能量图现在使用 `d => d.step` 作为 X 获取器，范围来自第一个和最后一个条目的步进编号：
```js
const xRange = { min: first.step, max: last.step };
```
由于缓冲区始终包含 `[step_N−299, …, step_N]`，范围每帧向前滑动 1，因此 X 轴标签持续变化。指标清除时重置为 0。

# 第九阶段 — 移动端与响应式优化

## 动机

该应用是在大桌面显示器上设计的。在手机上：
- 控制栏过于拥挤或溢出。
- 参数面板（重力/阻尼/速度滑块）占用了宝贵的垂直空间。
- 触控拖拽摆锤可能触发页面滚动而非移动摆锤。
- 1200 点的轨迹缓冲区在移动设备上造成不必要的 GPU 负载。

## 变更

### 触控事件加固

**问题**：现有的触控处理程序使用 `{ passive: true }`，这意味着浏览器将其视为被动监听器——`preventDefault()` 将被忽略。特别是在 iOS Safari 上，即使 `<body>` 设置了 `overflow: hidden`，用户尝试拖拽摆锤时页面仍然可能弹性滚动。

**修复**：
- 将 `touchstart` 和 `touchmove` 监听器均改为 `{ passive: false }`。
- 在两个处理程序内部，仅在拖拽激活（`dragActive === true`）时调用 `e.preventDefault()`。这在不干扰非拖拽触摸的情况下抑制了页面弹性滚动。
- 通过 CSS 在两个画布上添加 `touch-action: none`——这在合成器级别告诉浏览器永远不要为滚动/缩放拦截画布上的触摸手势。
- 在 `html, body` 上添加 `overscroll-behavior: none`，进一步阻止所有浏览器上的弹性弹跳效果。

### 性能 — 轨迹长度限制

**问题**：`TRAIL_LENGTH = 1200` 意味着每个粒子存储 1200 个 `{x, y, s}` 对象。有 N 个摆每个有 2–5 个粒子时，那就是数千个点每帧在 80 个批次中重新绘制。在移动 GPU 上这可能导致帧率下降。

**修复**：
- 通过 `window.matchMedia("(max-width: 768px)").matches` 检测移动设备。
- 移动端设置 `TRAIL_LENGTH` 为 600（一半的点，一半的绘制调用）。
- 从 `const` 改为 `let`，使该值可以在方向变化时调整（目前未实施，但无需重构即可实现）。

### 响应式 CSS 布局

**问题**：UI 是刚性的。固定宽度的元素如 330px 的图表框、160px 最小宽度的参数面板和 12px 间隙的控制栏假定视口 ≥1024px。

**修复**：一个单一的 `@media (max-width: 768px)` 块处理所有移动端调整：

| 元素 | 桌面端 | 移动端 |
|------|--------|--------|
| 控制栏间隙 | 12px | 6px |
| 控制栏按钮字号 | 12px | 9px |
| 角度显示字号 | 11px | 9px |
| 参数面板 | `display: flex`（始终可见） | `display: none`（齿轮切换） |
| 参数位置 | 右上 | 右下（控制栏上方） |
| 图表框 | 并排角落 | 堆叠居中（上 35% / 下 35%） |
| 图表框尺寸 | 固定 330×240 px | 90% 宽度，自动高度 |
| 上下文菜单位置 | 底部 105px | 底部 80px |

### 可折叠设置面板

**问题**：重力/阻尼/速度滑块很有用，但在手机上占用太多屏幕。不应完全移除——只需在需要时显示。

**修复**：
- 在控制栏中添加 `⚙` 齿轮按钮（Clear Trail 之后，Save 之前）。
- 齿轮按钮在桌面端隐藏，通过 `#btn-gear { display: none }`。
- 在移动端，通过媒体查询覆盖 `#btn-gear { display: inline-block !important }`。
- 点击齿轮调用 `toggleSettingsPanel()`，在 `#params-panel` 上切换 `.show` 类。
- 在移动端，参数面板位于 `bottom: 55px`（紧贴控制栏上方）并浮动到右侧。

## 变更文件

- **Plan.md**：新的第九阶段部分，包含任务和验收标准。
- **style.css**：画布上的 `touch-action:none`，根元素上的 `overscroll-behavior:none`，完整的 `@media (max-width: 768px)` 块，齿轮按钮显示规则。

### 缺陷修复：移除 slowMo 后启动时画布空白

移除 `slowMo` 变量和 `btn-slow` 按钮后，初始加载时摆体不可见——画布只显示暗色背景。

**根本原因**：两个独立的问题：

1. **缺少同步初始绘制**：引导序列为 `addPendulum()` → `resizeCanvas()` → `updateControls()` → `animate()`。`animate()` 函数仅通过 `requestAnimationFrame(animate)` 排队第一帧，约 16ms 后在下一个 VSync 触发。在此间隙中，画布是透明的（未绘制），只显示 body 的 `#0a0a0f` 背景。

2. **不安全的 `matchMedia` 使用**：`window.matchMedia("(max-width: 768px)").matches` 在无守卫的情况下调用。如果 `matchMedia` 不可用或抛出异常，整个顶层脚本将在该行停止——其后的所有代码都不会执行。

**修复**：
- 添加空守卫：`window.matchMedia ? window.matchMedia(...).matches : false`。
- 在引导过程中、`animate()` 之前添加了同步的 `draw()` 和 `updateAngleDisplay()` 调用——摆体在第一个布局帧上即渲染，而非等待第一个 rAF 回调。

### 缺陷修复：GitHub Pages 上"无法读取 null 的属性 'addEventListener'"错误

部署到 GitHub Pages 后，脚本在启动时崩溃，出现 `TypeError: Cannot read properties of null (reading 'addEventListener')`，错误指向某行 `document.getElementById(...).addEventListener(...)`。这导致整个动画循环从未启动——没有摆体、没有控制项，一个空白暗色页面。

**根本原因**：两个问题叠加：

1. **时机问题 — 脚本在 DOM 完全加载前执行**。`<script>` 标签没有 `defer` 属性，因此浏览器在 HTML 解析过程中遇到它时立即执行。如果网络以分块方式交付 HTML，脚本元素可能在后来的 DOM 节点被解析之前执行。在本地（file:// 或 live server），HTML 瞬间到达，因此这从未成为问题。在 GitHub Pages（CDN 交付、分块传输）上，时机窗口打开。

2. **缺失元素 — ID 不匹配**。某个 `document.getElementById()` 调用返回 `null`，因为具有该 ID 的元素在执行时不存在（要么尚未解析，要么 ID 在 HTML 中已重命名但 JS 中未更新）。对 `null` 调用 `.addEventListener()` 会抛出 TypeError，该异常从顶层脚本传播出去，停止所有后续代码。

**修复**：

- 在 `<script>` 标签上添加 `defer`，使浏览器始终等待完整 HTML 解析后才执行 JS。
- 添加 `$()` 辅助函数（`document.getElementById` 别名）以简化代码。
- 添加 `on(id, event, handler)` 双用途函数：
  - `on(id)` — 返回元素（类似 `$()`），用于直接访问。
  - `on(id, event, handler)` — 安全地绑定事件监听器。如果 `$(id)` 返回 `null`，则静默跳过。
- 将所有 18 处 `document.getElementById(id).addEventListener(event, handler)` 调用转换为 `on(id, event, handler)`。
- 保留 `window`、`document` 和 `canvasB` 的监听器不变（这些 DOM 节点始终存在）。

# 后期修复 — CSS 优先级冲突与应用内 Wiki 阅读器

## 修复 1：围栏代码块显示为一行

**报告**：在指南弹窗中查看完整 README 时，架构部分的目录树结构显示为单行，围栏代码块中的换行符未保留。

**根本原因**：`style.css` 中第 527 行的 `#help-modal-content code { white-space: nowrap }`。这个基于 ID 的选择器（优先级 1,0,0,1）适用于弹窗中**所有** `<code>` 元素，包括渲染后的 README 内容中 `<pre>` 块内部的代码。基于类的 `.help-readme-body pre code { white-space: pre }` 规则（优先级 0,2,0,2）被覆盖，因为 ID 选择器权重更高——CSS 优先级意味着 `#id` 无论顺序如何都高于 `.class`。

**为何重排序无效**：将 `.help-readme-body pre code` 规则移到 `.help-readme-body code` 之后在样式表中没有帮助，因为前者被 ID 选择器击败。CSS 优先级是一个权重系统，而非顺序系统。

**修复**：在 `pre code` 选择器中添加父级 ID 以匹配优先级：
```css
.help-readme-body pre code,
#help-modal-content .help-readme-body pre code {
    white-space: pre;
    /* … */
}
```
`#help-modal-content .help-readme-body pre code` 的优先级为 1,1,2,1——它胜过了基础的 `#help-modal-content code`（1,0,0,1）。

**影响范围**：自动应用于任何 `.help-readme-body` 内渲染的所有围栏代码块，包括 `AI_DISCLOSURE.md`/`AI_DISCLOSURE_ZH.md` 中的协作工作流章节。

## 修复 2：应用内 Wiki 阅读器 — 拦截 Markdown 链接

**报告**：`README.md` 包含指向其他 `.md` 文件的相对链接（如 `AI_DISCLOSURE.md`）。在指南弹窗中点击这些链接会触发默认浏览器导航（404 或文件下载），破坏了应用内阅读体验。

### 解决方案：Wiki 导航

三项更改将指南弹窗转变为递归 wiki 阅读器：

**a. 通用化的 `loadAndRenderMD(path)`** — 替换了之前硬编码为 `./README.md` 的 `fetchAndShowReadme()`。新函数：
- 接受任意 `.md` 相对路径，通过 `fetch()` 获取，使用现有的 `parseSimpleMarkdown()` 解析，并将 HTML 渲染到 `#help-body` 中。
- 在 `currentDocPath` 中跟踪当前文档。
- 自动前置返回导航栏：根 README 显示"← 返回操作说明"，子页面显示"← 返回指南"。

**b. `bindMarkdownLinks()`** — 每次渲染后调用。扫描 `.help-readme-body` 内的所有 `<a>` 标签，为任何 `href` 以 `.md` 结尾的链接附加点击处理程序。处理程序调用 `e.preventDefault()` 和 `loadAndRenderMD(href)`，在应用内加载链接文档。

**c. 链接渲染** — `processInline()` 现在区分 `.md` 链接和外部链接。`.md` 链接渲染时不加 `target="_blank"`（因为它们在内部导航）；所有其他链接保留 `target="_blank"` 以确保安全。

### 导航流程
```
控制表 → 点击"查看完整指南 →" → README.md
     ↑                                  ↓
  "← 返回操作说明"           点击 [AI 公开声明] 链接
     ↑                                  ↓
     └────────── README.md ◄── "← 返回指南" ── AI_DISCLOSURE.md
```

### 变更文件
- **script.js**：约 70 行变更。添加了 `currentDocPath` 跟踪、`loadAndRenderMD()`、`bindMarkdownLinks()`、`processInline()` 中的条件 `target="_blank"`、事件委托中的 `.help-back-guide` 处理程序。中英文各两个新的 i18n 键。
- **style.css**：`.help-back-guide` 与 `.help-back-btn` 共享样式。

### 边界情况
- 子页面上的链接（如 `AI_DISCLOSURE.md` → `AI_DISCLOSURE_ZH.md`）会被递归拦截——链接链可工作到任意深度。
- 加载错误显示错误消息；重新打开弹窗时返回控制总览。
- 语言切换重置为控制总览（弹窗切换间不持久化状态）。

# 功能 — 状态序列化、URL 分享与恢复

## 动机
用户希望能够通过 URL 与他人分享特定的摆配置。不允许使用外部库——仅限原生 Web API。

## 实现

### 序列化格式
使用简洁的单字母键名以缩短 URL：
```json
{"v":1,"g":11,"d":0.0003,"s":1,"p":[{"n":2,"t":[135,135],"c":0,"v":true}]}
```
`v` = 版本号, `g` = 重力, `d` = 阻尼, `s` = 速度倍率；每个摆包含 `n` = 连接杆数, `t` = 角度（度）, `c` = 调色板索引, `v` = 可见性。

### URL 安全 Base64
使用 `encodeURIComponent` + `btoa` 处理 UTF-8 字符（颜色十六进制码、中文标签、表情符号），防止 `btoa()` 在非 Latin-1 字符上抛出异常。

### 自动保存
`saveStateToURL()` 在 10 个状态变更点被调用：滑块输入、添加/删除摆、换色、可见性切换、关节增减、重置、混沌模式切换、拖拽结束。

### 分享按钮
点击 → 复制当前 URL（含 `#state=...`）到剪贴板 → 按钮显示 "✓ 已复制!" 反馈 1.8 秒。

### 状态恢复
页面初始化时检查 `window.location.hash`。解码 Base64，验证每个字段（`isFinite()` 检查 + 范围钳制），重构摆和物理参数。失败时优雅降级到默认设置。

### 验证结果
- ✅ 含状态哈希的页面加载正确恢复 2 个摆、5+3 节、不同角度和颜色
- ✅ 全局参数（重力、阻尼、速度）正确恢复
- ✅ 隐藏的摆正确恢复为不可见
- ✅ 损坏的数据优雅降级，无崩溃

# 功能 — 拖拽极坐标网格 HUD

## 动机
拖拽摆锤设置角度时有"磁吸对齐"功能（15°增量），但没有任何视觉反馈显示对齐位置。用户需要猜测摆锤最终位置。

## 设计
在活动拖拽期间，在 Layer B（摆锤画布）上渲染一个临时的极坐标网格/量角器 HUD 叠加层。

### 状态变量
```js
let dragHudOpacity = 0;             // 当前透明度，动画过渡 0 ↔ 0.3
const dragHudPivot = { x: 0, y: 0 }; // 被拖拽摆锤的父支点（缓存）
let dragHudRadius = 0;               // 杆长（像素，缓存）
```

### 拖拽起始时的数据捕获
`mousedown` 和 `touchstart` 处理程序在拖拽开始时捕获父支点坐标和杆长，存入 `dragHudPivot`/`dragHudRadius`。缓存的数值在释放后的淡出动画中持续使用。

### 拖拽期间的实时刷新
`animate()` 循环在 `dragActive === true` 时每帧从当前粒子位置刷新 `dragHudPivot`，确保窗口缩放不影响 HUD 位置。

### 透明度动画
```js
// 淡入（拖拽中）：
dragHudOpacity = Math.min(dragHudOpacity + 0.05, 0.3);
// 淡出（释放后）：
dragHudOpacity = Math.max(dragHudOpacity - 0.15, 0);
```
～125 毫秒淡入，～33 毫秒淡出。

### 绘制（`drawDragHUD()`）

在单个 `ctxB.save/restore` 块中绘制三层，全局 alpha = `dragHudOpacity`：

1. **杆长圆** — 虚线（`setLineDash([5, 5])`），`rgba(255,255,255,0.55)`，1.2px 线宽，以支点为中心、杆长为半径。显示摆锤运动的轨迹弧线。

2. **15° 径向线** — 从支点向外 24 条线，`setLineDash([2, 4])`，`rgba(255,255,255,0.28)`，0.8px 线宽。使用 `sin()` 计算 X、`cos()` 计算 Y，与物理引擎的角度约定（`atan2(dx, dy)`，0 = 正下方）一致。

3. **对齐指示点** — 在弧线上当前 `snapAngle(thetas[dragTarget - 1])` 位置绘制亮白点（4px 半径），带 8px 半透明外圈。仅在活动拖拽期间显示。

### 集成点
| 位置 | 操作 |
|---|---|
| `canvasB mousedown` | 捕获 HUD 数据，设置 `opacity = 0.01` |
| `canvasB touchstart` | 触屏对应的捕获操作 |
| `animate()` 起始处 | 实时刷新支点位置，淡入/淡出透明度 |
| `draw()` 摆锤循环后 | 在绘制枢轴点前调用 `drawDragHUD()` |

### 边界情况
- **拖拽中窗口缩放**：每帧从当前粒子位置实时刷新 `dragHudPivot`，缓存仅用于淡出阶段。
- **多摆系统**：支持任意选中摆；通过 `selectedPendulum` 和 `dragTarget` 确定使用哪个链接（link）的支点和半径。
- **触屏**：`touchstart` 中同样的捕获逻辑，淡入淡出动画与鼠标一致。
- **零透明度保护**：`drawDragHUD()` 在 `dragHudOpacity <= 0.001` 时立即返回。

## 文件变更
- **script.js**：新增 88 行。新状态变量 `dragHudOpacity`、`dragHudPivot`、`dragHudRadius`。新函数 `drawDragHUD()`。修改 `mousedown`、`touchstart`、`draw()` 和 `animate()`。
